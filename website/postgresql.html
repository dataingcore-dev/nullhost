PostgreSQL متقدم — الاستعلامات والمؤشرات والأداء

SQL متقدم: JOINs, CTEs, indexing, optimization مع بايثون

← العودة إلى الرئيسية

هذا الفصل يغطي PostgreSQL بشكل متقدم — الاستعلامات المعقدة، تحسين الأداء، والتفاعل مع بايثون عبر psycopg2 و SQLAlchemy.

1. JOINs المتقدمة

الـ JOIN هو أداة ربط بين جدولين أو أكثر بناءً على علاقة بينهما. في PostgreSQL، لديك عدة أنواع من JOIN تتيح تحكمًا دقيقًا في البيانات التي تريد جلبها. هذا المفهوم أساسي لأي تطبيق يستخدم قاعدة بيانات علائقية.

-- INNER JOIN — السجلات المشتركة فقط SELECT users.name, orders.total FROM users INNER JOIN orders ON users.id = orders.user_id; -- LEFT JOIN — كل المستخدمين حتى من دون طلبات SELECT users.name, COUNT(orders.id) AS order_count FROM users LEFT JOIN orders ON users.id = orders.user_id GROUP BY users.name; -- JOIN مع جداول متعددة SELECT u.name, p.title, o.quantity FROM orders o JOIN users u ON o.user_id = u.id JOIN products p ON o.product_id = p.id WHERE o.status = 'completed';
🔍 كيف يعمل؟ INNER JOIN يُرجع فقط الصفوف المتطابقة في كلا الجدولين. LEFT JOIN يُرجع كل صفوف الجدول الأيسر (users) حتى لو لم يكن هناك تطابق في orders. JOIN مع جداول متعددة تربط 3 جداول: orders → users عبر user_id، و orders → products عبر product_id. النتيجة تعرض فقط الطلبات المكتملة.

2. CTE — Common Table Expressions

CTE (Common Table Expression) تسمح لك بتعريف استعلام مؤقت باسم يمكنك استخدامه داخل استعلام أكبر. هذا يجعل الكود أكثر readability وقابلية للصيانة، خاصة مع الاستعلامات المعقدة التي تحتاج خطوات متعددة.

-- WITH: استعلام مساعد قبل الاستعلام الرئيسي WITH user_orders AS ( SELECT user_id, COUNT(*) AS total_orders, SUM(total) AS total_spent FROM orders GROUP BY user_id ) SELECT u.name, uo.total_orders, uo.total_spent FROM users u JOIN user_orders uo ON u.id = uo.user_id ORDER BY uo.total_spent DESC; -- CTE متكرر (Recursive) — التسلسل الهرمي WITH RECURSIVE org_tree AS ( SELECT id, name, manager_id, 1 AS level FROM employees WHERE manager_id IS NULL UNION ALL SELECT e.id, e.name, e.manager_id, ot.level + 1 FROM employees e JOIN org_tree ot ON e.manager_id = ot.id ) SELECT * FROM org_tree;
🔍 كيف يعمل؟ CTE الأول يُنشئ جدولاً مؤقتاً باسم user_orders يحسب عدد الطلبات وإجمالي الإنفاق لكل مستخدم. الاستعلام الرئيسي يربطه مع جدول users ويعرض النتائج مرتبة حسب أعلى إنفاق. CTE المتكرر (RECURSIVE) يبني هيكلاً تنظيمياً: يبدأ بالمديرين (حيث manager_id IS NULL)، ثم يضيف الموظفين الذين يتبعونهم، ويتكرر حتى يصل لكل الموظفين.

3. Indexing — المؤشرات للأداء

المؤشرات (Indexes) هي هيكل بيانات يُسرّع استعلامات البحث بشكل هائل. بدون index، تقوم PostgreSQL بمسح الجدول كاملاً (Seq Scan) — ومع index، تذهب مباشرة للصفحة المطلوبة. الفرق قد يكون من ثوانٍ إلى أجزاء من الألف من الثانية.

-- مؤشر عادي — B-tree (افتراضي) CREATE INDEX idx_users_email ON users(email); -- مؤشر مركب (Composite) — لاستعلامات متعددة الأعمدة CREATE INDEX idx_orders_user_status ON orders(user_id, status); -- مؤشر جزئي (Partial) — فقط لصفوف محددة CREATE INDEX idx_active_orders ON orders(created_at) WHERE status = 'active'; -- مؤشر فريد (Unique) CREATE UNIQUE INDEX idx_users_username ON users(username); -- تحليل الاستعلامات EXPLAIN ANALYZE SELECT * FROM users WHERE email = 'test@test.com';
🔍 كيف يعمل؟ المؤشر العادي (B-tree) يُسرّع البحث والترتيب. المؤشر المركب على (user_id, status) يُسرّع الاستعلامات التي تُصفّي بـ user_id و status معاً. المؤشر الجزئي يُؤرشف فقط الصفوف النشطة — أصغر حجماً وأسرع. المؤشر الفريد يمنع تكرار القيم. استخدم EXPLAIN ANALYZE لترى خطة تنفيذ الاستعلام والوقت الفعلي — سترى الفرق بين Seq Scan و Index Scan.

4. Optimization — تحسين الأداء

تحسين أداء قاعدة البيانات مهارة أساسية. مع نمو بياناتك، الاستعلامات البطيئة تصبح عنق الزجاجة. إليك الممارسات الأساسية لتحسين PostgreSQL:

-- 1. استخدم LIMIT بدلاً من جلب كل البيانات SELECT * FROM logs ORDER BY created_at DESC LIMIT 100; -- 2. تجنب SELECT * في الإنتاج SELECT id, name, email FROM users; -- 3. استخدم VACUUM لتنظيف الصفوف المحذوفة VACUUM ANALYZE orders; -- 4. Partitioning — تقسيم الجداول الكبيرة CREATE TABLE orders_2024 PARTITION OF orders FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');
⚠️ خطأ شائع: استخدام SELECT * في الإنتاج يجلب أعمدة غير ضرورية ويمنع تحسين الفهرس. حدّد الأعمدة التي تحتاجها فقط. أيضاً، لا تنسَ تشغيل VACUUM ANALYZE دورياً لتنظيف الصفوف المحذوفة وتحديث الإحصائيات — هذا يحافظ على أداء الاستعلامات.

5. PostgreSQL مع Python (psycopg2)

psycopg2 هو أشهر مكتبة بايثون للاتصال بقاعدة بيانات PostgreSQL. تدعم الاستعلامات المParametrized (آمنة من SQL Injection)، و Context Managers، والإدراج المتعدد (batch insert). إليك النمط الاحترافي:

import psycopg2 from psycopg2.extras import RealDictCursor, execute_values from contextlib import contextmanager DATABASE_URL = "postgresql://user:pass@localhost:5432/mydb" @contextmanager def get_db(): conn = psycopg2.connect(DATABASE_URL, cursor_factory=RealDictCursor) try: yield conn conn.commit() except Exception: conn.rollback() raise finally: conn.close() # استعلام مع (parameterized query — آمن من SQL injection) def get_user(user_id: int): with get_db() as conn: cursor = conn.cursor() cursor.execute("SELECT * FROM users WHERE id = %s", (user_id,)) return cursor.fetchone() # إدراج متعدد (batch insert) def create_users(users: List[dict]): with get_db() as conn: cursor = conn.cursor() execute_values( cursor, "INSERT INTO users (name, email) VALUES %s", [(u['name'], u['email']) for u in users] )
🔍 كيف يعمل؟ نستخدم @contextmanager لإنشاء مدير سياق يتعامل مع الاتصال بقاعدة البيانات: يُنشئ الاتصال، يُسلّمه للدالة (yield)، ثم يُغلق الاتصال تلقائياً بعد الانتهاء. RealDictCursor يجعل النتائج كـ dict بدلاً من tuple — أسهل للتعامل. execute_values تُدرج بيانات متعددة دفعة واحدة (batch insert) — أسرع بكثير من INSERT لكل صف.

6. SQLAlchemy — ORM احترافي

SQLAlchemy هو ORM (Object Relational Mapper) يسمح لك بالتعامل مع قاعدة البيانات باستخدام كائنات بايثون بدلاً من كتابة SQL يدوياً. هذا يسرّع التطوير ويجعل الكود أنظف، خاصة في المشاريع الكبيرة.

# تثبيت: pip install sqlalchemy psycopg2-binary from sqlalchemy import create_engine, Column, Integer, String, Float, ForeignKey from sqlalchemy.orm import declarative_base, sessionmaker, relationship engine = create_engine(DATABASE_URL) Base = declarative_base() SessionLocal = sessionmaker(bind=engine) class User(Base): __tablename__ = 'users' id = Column(Integer, primary_key=True) name = Column(String(100), nullable=False) email = Column(String(100), unique=True) orders = relationship("Order", back_populates="user") class Order(Base): __tablename__ = 'orders' id = Column(Integer, primary_key=True) total = Column(Float) user_id = Column(Integer, ForeignKey('users.id')) user = relationship("User", back_populates="orders") Base.metadata.create_all(engine) # استخدام db = SessionLocal() users = db.query(User).filter(User.email == "test@test.com").all() db.close()
🔍 كيف يعمل؟ SQLAlchemy ORM: first نعرف النماذج (Models) ككلاس بايثون — كل كلاس يمثل جدولاً. الأعمدة تُعرّف كـ Column بأنواعها. relationship تُنشئ علاقة بين الجدولين (user له orders متعددة، order ينتمي لـ user). create_all يُنشئ الجداول تلقائياً. SessionLocal تُنشئ جلسة للتعامل مع قاعدة البيانات بشكل كائني.

📝 ملخص PostgreSQL

  • CTE (WITH): تبسيط الاستعلامات المعقدة
  • Indexes: تحسين أداء البحث (B-tree, Partial, Composite)
  • EXPLAIN ANALYZE: تحليل أداء الاستعلامات
  • psycopg2: المكتبة الأساسية للتعامل مع PostgreSQL من بايثون
  • SQLAlchemy: ORM قوي يسهل التعامل مع قاعدة البيانات
