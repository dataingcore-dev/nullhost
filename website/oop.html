البرمجة كائنية التوجه OOP في بايثون

دليل متكامل وشامل: كلاسات، وراثة، بوليمورفيزم، Dunder Methods، Decorators، ومبادئ SOLID

1. مقدمة عن البرمجة كائنية التوجه (OOP)

البرمجة كائنية التوجه (Object-Oriented Programming) هي نموذج برمجي يعتمد على مفهوم "الكائنات" (Objects) التي تحتوي على بيانات (خصائص/Attributes) وسلوك (طرق/Methods). بايثون هي لغة برمجة كائنية التوجه بالكامل - كل شيء فيها هو كائن.

مبدأ أساسي: OOP يساعدك على تنظيم الكود وجعله أكثر قابلية لإعادة الاستخدام والصيانة. كلما كبر مشروعك، كلما كانت الحاجة إلى OOP أكبر.

2. الكلاسات (Classes) - نظرة عميقة

الـ Class هو المخطط (Blueprint) الذي يُنشئ منه الكائنات. دعنا نرى كلاس متقدم مع جميع العناصر المهمة:

class Employee: # Class attribute - مشترك بين جميع النسخ raise_amount = 1.04 num_of_emps = 0 def __init__(self, first, last, pay): # Instance attributes - خاصة بكل نسخة self.first = first self.last = last self.pay = pay self.email = first + '.' + last + '@company.com' Employee.num_of_emps += 1 # Regular method - تأخذ self كأول باراميتر def fullname(self): return f"{self.first} {self.last}" def apply_raise(self): self.pay = int(self.pay * self.raise_amount) # Classmethod - تأخذ cls كأول باراميتر @classmethod def set_raise_amount(cls, amount): cls.raise_amount = amount @classmethod def from_string(cls, emp_str): # Alternative constructor first, last, pay = emp_str.split('-') return cls(first, last, int(pay)) # Staticmethod - لا تأخذ self ولا cls @staticmethod def is_workday(day): return day.weekday() != 5 and day.weekday() != 6
🔍 شرح الكلاس Employee وأنواع الميثودز

هذا الكلاس يمثل نموذجاً متكاملاً لموظف ويوضح الفروقات الجوهرية بين أنواع الميثودز الثلاثة في بايثون. Class attributes مثل raise_amount هي خصائص مشتركة بين جميع نسخ الكلاس ويمكن الوصول إليها عبر Employee.raise_amount أو self.raise_amount. Instance methods مثل fullname و apply_raise تأخذ self كأول وسيط وتتعامل مع بيانات كل كائن على حدة. Class methods المزينة بـ @classmethod تأخذ cls وتستخدم بشكل أساسي كـ alternative constructors أو لتعديل خصائص الكلاس نفسه. أما Static methods المزينة بـ @staticmethod فهي دوال عادية توضع داخل الكلاس لأسباب تنظيمية ولا تحتاج للوصول إلى الكلاس أو الكائن. الخاصية num_of_emps هي class attribute تزداد تلقائياً مع إنشاء كل كائن جديد، مما يتيح تتبع عدد الموظفين.

أنواع الطرق (Methods)

  • Instance Methods (طرق النسخة): تأخذ self كأول وسيط. يمكنها الوصول وتعديل بيانات الكائن.
  • Class Methods (طرق الكلاس): تأخذ cls كأول وسيط (ديكوريتور @classmethod). تستخدم لإنشاء alternative constructors أو تعديل خصائص الكلاس.
  • Static Methods (طرق ثابتة): لا تأخذ self ولا cls (ديكوريتور @staticmethod). دوال عادية داخل الكلاس لأسباب تنظيمية.
# استخدام classmethod كـ alternative constructor emp_str_1 = 'John-Doe-70000' new_emp_1 = Employee.from_string(emp_str_1) print(new_emp_1.email) # John.Doe@company.com # استخدام staticmethod import datetime my_date = datetime.date(2024, 5, 10) print(Employee.is_workday(my_date)) # True (Friday)
🔍 تطبيق classmethod و staticmethod عملياً

هذا المثال يوضح كيف يمكن استخدام @classmethod كـ alternative constructor عبر الميثود from_string التي تستقبل نصاً بتنسيق "الاسم-اللقب-الراتب" وتحوله إلى كائن Employee. هذا النمط شائع جداً في مكتبات بايثون مثل datetime الذي يوفر constructors بديلة مثل fromtimestamp و fromisoformat. أما @staticmethod is_workday فتحدد إذا كان اليوم المعطى يوم عمل (الإثنين إلى الجمعة) باستخدام دالة weekday() التي تعيد 5 للسبت و 6 للأحد. static methods تكون مفيدة للدوال المساعدة التي لها علاقة منطقية بالكلاس لكنها لا تحتاج للوصول إلى self أو cls.

@property Decorator - الخصائص المحسوبة

يسمح لك بتعريف methods يتم الوصول إليها كأنها attributes:

class Employee: def __init__(self, first, last): self.first = first self.last = last @property def fullname(self): return f"{self.first} {self.last}" @property def email(self): return f"{self.first}.{self.last}@company.com" @fullname.setter def fullname(self, name): first, last = name.split(' ') self.first = first self.last = last @fullname.deleter def fullname(self): self.first = None self.last = None # الاستخدام emp = Employee('John', 'Smith') print(emp.fullname) # John Smith - بدون () لأنها property print(emp.email) # John.Smith@company.com emp.fullname = 'Corey Schafer' # يستخدم setter del emp.fullname # يستخدم deleter
🔍 @property Decorator ودوره في التحكم بالخصائص

الديكوريتور @property يحول دالة إلى خاصية readable يمكن الوصول إليها بدون أقواس ()، مما يجعل الكود أكثر نظافة واتساقاً. يتيح لك @property تعريف getter يحسب القيمة ديناميكياً (مثل fullname الذي يدمج first و last). باستخدام @fullname.setter يمكنك التحكم في عملية التعيين (assignment) وتنفيذ منطق validation أو side effects عند تغيير القيمة. @fullname.deleter يتحكم في سلوك del obj.attribute. هذه الآلية تسمح لك بالحفاظ على واجهة بسيطة (تبدو كأنها attribute عادي) مع إخفاء منطق معقد خلف الكواليس، دون الحاجة لكتابة getter و setter صريحين كما في لغات مثل Java.

3. الوراثة (Inheritance) - مستوى احترافي

الوراثة تسمح لكلاس باشتقاق خصائص وسلوك كلاس آخر. هذا يقلل تكرار الكود ويسهل الصيانة.

وراثة فردية (Single Inheritance)

class Developer(Employee): raise_amount = 1.10 # تجاوز قيمة الكلاس الأب def __init__(self, first, last, pay, prog_lang): super().__init__(first, last, pay) # استدعاء __init__ الأب self.prog_lang = prog_lang dev_1 = Developer('Sara', 'Ali', 80000, 'Python') print(dev_1.email) # Sara.Ali@company.com print(dev_1.prog_lang) # Python print(dev_1.pay) # 80000 dev_1.apply_raise() print(dev_1.pay) # 88000 (raise_amount = 1.10)
🔍 الوراثة الفردية واستخدام super()

كلاس Developer يرث من Employee ويكتسب تلقائياً جميع الخصائص والميثودز مثل email و fullname و apply_raise. يستخدم super().__init__(first, last, pay) لاستدعاء constructor الكلاس الأب وتجنب تكرار كود التهيئة، مع إضافة الخاصية الجديدة prog_lang. تم تغيير raise_amount إلى 1.10 في الكلاس الابن، مما يوضح concept الـ override حيث أن الكلاس الابن يمكنه تجاوز قيم الكلاس الأب. هذا المبدأ يسمى DRY (Don't Repeat Yourself) حيث نعيد استخدام كود الأب ونضيف فقط الجديد في الابن.

super(): تسمح باستدعاء methods من الكلاس الأب. هذا يمنع تكرار الكود ويحافظ على مبدأ DRY (Don't Repeat Yourself).

تسلسل البحث عن الميثود (MRO - Method Resolution Order)

MRO هو الترتيب الذي تبحث فيه بايثون عن method في سلسلة الوراثة:

print(Developer.__mro__) # (, , ) # يمكنك رؤية MRO باستخدام: print(Developer.mro()) # نفس النتيجة print(help(Developer)) # يظهر MRO مع كل التفاصيل
🔍 MRO - Method Resolution Order

MRO هو الترتيب الذي تتبعه بايثون عند البحث عن method في هرم الوراثة، وتستخدم خوارزمية C3 Linearization لحسابه. يمكنك عرض MRO عبر ClassName.__mro__ أو ClassName.mro(). في الوراثة الفردية، MRO يكون بسيطاً: الكلاس الابن → الكلاس الأب → object. فهم MRO ضروري جداً في الوراثة المتعددة لتوقع أي method سيتم استدعاؤه. عندما تستدعي apply_raise() على كائن Developer، تبحث بايثون أولاً في Developer، ثم Employee، وأخيراً object.

الوراثة المتعددة (Multiple Inheritance)

بايثون تدعم الوراثة من عدة كلاسات. الترتيب يهم ويحدد MRO:

class Manager(Employee): def __init__(self, first, last, pay, employees=None): super().__init__(first, last, pay) if employees is None: self.employees = [] else: self.employees = employees def add_emp(self, emp): if emp not in self.employees: self.employees.append(emp) def remove_emp(self, emp): if emp in self.employees: self.employees.remove(emp) def print_emps(self): for emp in self.employees: print(f'--> {emp.fullname()}') # مثال الوراثة المتعددة class A: def do_something(self): return 'Method from A' class B: def do_something(self): return 'Method from B' class C(A, B): # A له الأولوية (C3 Linearization) pass c = C() print(c.do_something()) # 'Method from A' print(C.__mro__) # C -> A -> B -> object
🔍 الوراثة المتعددة وأولوية MRO

كلاس Manager يرث من Employee ويضيف قائمة بالموظفين التابعين له مع ميثودز لإدارة هذه القائمة مثل add_emp و remove_emp. في الوراثة المتعددة (كلاس C(A, B))، ترتيب الكلاسات الأب في تعريف الكلاس هو الذي يحدد الأولوية: الكلاس المذكور أولاً (A) له الأولوية عند البحث عن الميثودز. خوارزمية C3 Linearization تضمن أن الترتيب يحافظ على اتساق: الكلاس الابن قبل الأب، وإذا ورث من عدة كلاسات فالترتيب في definition يحدد الأولوية. MRO لـ C هو: C → A → B → object، لذلك c.do_something() ستعيد 'Method from A'.

Mixins - إعادة استخدام السلوك

Mixins كلاسات صغيرة توفر functionality محددة، تُستخدم مع الوراثة المتعددة:

class LogMixin: def log(self, message): print(f'[LOG] {message}') class JSONMixin: def to_json(self): import json return json.dumps(self.__dict__) class User(LogMixin, JSONMixin): def __init__(self, name): self.name = name self.log(f'User {name} created') user = User('Ahmed') print(user.to_json()) # {"name": "Ahmed"}
🔍 Mixins - إعادة استخدام السلوك بمرونة

Mixins هي كلاسات صغيرة ومتخصصة توفر functionality محددة ولا تُستخدم بمفردها أبداً. LogMixin يوفر capability التسجيل (log) بينما JSONMixin يوفر تحويل الكائن إلى JSON عبر to_json. كلاس User يرث من كلا الميكسينز ويكتسب قدراتهما دون الحاجة لوراثة هرمية معقدة. هذه التقنية مفيدة جداً لإضافة capabilities متعددة (مثل logging, serialization, validation) إلى كلاسات متعددة دون تكرار كود. Mixins تتبع مبدأ "مسؤولية واحدة" حيث كل mixin يركز على وظيفة محددة.

Abstract Base Classes (ABC) - تقنيات متقدمة

تستخدم لفرض تنفيذ methods معينة في الكلاسات المشتقة. توفر بايثون آليات متعددة:

from abc import ABC, abstractmethod, ABCMeta # الطريقة 1: وراثة ABC (الأسهل والأشيع) class Shape(ABC): @abstractmethod def area(self): pass @abstractmethod def perimeter(self): pass @property @abstractmethod # Abstract property من Python 3.3 def description(self): pass class Square(Shape): def __init__(self, side): self.side = side def area(self): return self.side ** 2 def perimeter(self): return self.side * 4 @property def description(self): return f'Square(side={self.side})' # shape = Shape() # TypeError! لا يمكن إنشاء كائن من ABC square = Square(5) print(square.area()) # 25
🔍 Abstract Base Classes (ABC) وفرض الواجهات

ABCs تسمح لك بتعريف واجهات (interfaces) تجبر الكلاسات المشتقة على تنفيذ ميثودز محددة. الميثودز المزينة بـ @abstractmethod يجب أن تُنفذ في أي كلاس غير abstract يرث من Shape. إذا حاولت إنشاء كائن من Shape مباشرة، ستحصل على TypeError. يمكن أيضاً استخدام @abstractmethod مع @property كما في description لفرض تنفيذ خصائص معينة. هذا يضمن أن كل الأشكال الهندسية ستوفر واجهة موحدة (area, perimeter, description) مع تنفيذ مختلف لكل منها.

# الطريقة 2: metaclass=ABCMeta للكلاسات التي ترث من غير ABC class MyAbstract(metaclass=ABCMeta): @abstractmethod def must_implement(self): pass # Virtual Subclass - تسجيل كلاس بدون وراثة! @Shape.register class Circle: # لا يرث من Shape ولا يعلم بوجوده! def __init__(self, radius): self.radius = radius def area(self): return 3.14 * self.radius ** 2 def perimeter(self): return 2 * 3.14 * self.radius @property def description(self): return f'Circle(r={self.radius})' print(isinstance(Circle(3), Shape)) # True رغم أن Circle لا يرث! print(issubclass(Circle, Shape)) # True
🔍 Virtual Subclasses و ABCMeta

بالإضافة إلى ABC، يمكنك استخدام metaclass=ABCMeta لتعريف كلاسات abstract خاصة بتلك التي تحتاج لوراثة كلاس آخر غير ABC. الميزة الأقوى هي Virtual Subclasses عبر @Shape.register: يمكنك تسجيل أي كلاس كـ subclass للكلاس abstract بدون أن يرث منه، ولا يحتاج الكلاس المسجل حتى أن يعلم بوجود الكلاس abstract! هذا مفيد جداً عند التعامل مع مكتبات خارجية لا يمكنك تعديل كودها. بعد التسجيل، تعمل isinstance() و issubclass() بشكل طبيعي.

متى تستخدم register(): مع كلاسات من مكتبات خارجية لا يمكنك تعديلها. مثلاً: Iterator.register(MyCustomIter).

4. تعدد الأشكال (Polymorphism)

تعدد الأشكال يعني أن الكائنات المختلفة يمكنها الاستجابة لنفس الاستدعاء بطريقتها الخاصة:

Duck Typing

"إذا مشى مثل البطة وصوت مثل البطة، فهي بطة". بايثون تهتم بالسلوك وليس النوع:

class Dog: def speak(self): return 'Woof!' class Cat: def speak(self): return 'Meow!' class Duck: def speak(self): return 'Quack!' def make_sound(animal): print(animal.speak()) # لا يهم النوع، المهم وجود method speak # نفس الدالة تعمل مع أنواع مختلفة! make_sound(Dog()) # Woof! make_sound(Cat()) # Meow! make_sound(Duck()) # Quack!
🔍 Duck Typing - التصنيف حسب السلوك

Duck Typing هو جوهر تعدد الأشكال في بايثون: الدالة make_sound تهتم فقط بوجود الميثود speak() وليس بنوع الكائن. لا تحتاج الكلاسات Dog و Cat و Duck لوراثة كلاس مشترك أو تنفيذ واجهة محددة - يكفي أن توفر الميثود المطلوب. هذا يمنح مرونة هائلة ويقلل من التعقيد مقارنة باللغات الأخرى التي تفرض typing صارم. مع ذلك، يجب توثيق الواجهات المتوقعة جيداً لأن الخطأ سيكون في وقت التشغيل (runtime) وليس في وقت الترجمة.

Protocols - التصنيف الهيكلي (Structural Subtyping)

منذ Python 3.8، typing.Protocol يسمح بالتصنيف الهيكلي: "إذا طبقت الواجهة فأنت من النوع" - بدون وراثة صريحة:

from typing import Protocol, runtime_checkable @runtime_checkable # يسمح باستخدام isinstance() class Drawable(Protocol): def draw(self) -> str: ... # توقعات بدون تنفيذ (ellipsis) # هذه الكلاسات لا تعلم بوجود Drawable أصلاً! class Circle: def draw(self) -> str: return 'Drawing a circle' class Square: def draw(self) -> str: return 'Drawing a square' class Car: def drive(self) -> str: return 'Driving' def render(obj: Drawable): print(obj.draw()) render(Circle()) # صحيح - Circle تطابق Drawable هيكلياً render(Square()) # صحيح # render(Car()) # خطأ type checker - Car لا تملك draw() print(isinstance(Circle(), Drawable)) # True بفضل @runtime_checkable print(isinstance(Car(), Drawable)) # False
🔍 Protocols - التصنيف الهيكلي في Python 3.8+

Protocols توفر طريقة للتصنيف الهيكلي (structural subtyping) بدلاً من التصنيف الاسمي (nominal subtyping). الفرق الأساسي أن الكلاسات Circle و Square لا تحتاج لوراثة Drawable أو حتى معرفة وجوده - يكفي أن تطابق واجهته (تملك ميثود draw بتوقيع مناسب). @runtime_checkable يجعل الـ protocol قابلاً للاستخدام مع isinstance(). Protocols مفيدة جداً في أنظمة typing حيث تريد تعريف واجهات دقيقة مع الحفاظ على مرونة Duck Typing، وهي مفضلة على ABCs عند التعامل مع مكتبات خارجية.

ABC vs Protocol: ABC → "يجب أن ترث مني" (nominal subtyping). Protocol → "يجب أن تطابق واجهتي" (structural subtyping). استخدم Protocol مع مكتبات خارجية لا يمكنك تعديلها.

Protocols المدمجة في بايثون

بايثون تستخدم protocols في كل مكان - هذه آلية عمل الدوال المدمجة:

# Iterator Protocol (الموجود منذ البداية) class MyRange: def __init__(self, start, end): self.value = start self.end = end def __iter__(self): return self def __next__(self): if self.value >= self.end: raise StopIteration current = self.value; self.value += 1 return current for num in MyRange(1, 5): print(num) # 1 2 3 4 # Iterable = Protocol(__iter__) # Iterator = Protocol(__iter__, __next__) # Sized = Protocol(__len__) # Hashable = Protocol(__hash__)
🔍 Protocols المدمجة: Iterator Protocol

هذا مثال على كيفية عمل Iterator Protocol في بايثون. أي كلاس يطبق __iter__ (يعيد نفسه) و __next__ (يعيد القيمة التالية أو يرفع StopIteration) يصبح iterator قابلاً للاستخدام في حلقات for. كلاس MyRange يطبق هذا الـ protocol ويعمل تماماً مثل دالة range() المدمجة. بايثون تحتوي على العديد من protocols المدمجة: Iterable (__iter__)، Iterator (__iter__ + __next__)، Sized (__len__)، Hashable (__hash__)، وغيرها. هذه الـ protocols هي ما يجعل الدوال المدمجة مثل len() و iter() تعمل مع أي نوع مخصص.

5. Dunder Methods (Magic Methods) - دليل كامل

Dunder (Double UNDERscore) methods تسمح بتخصيص سلوك الكلاس ليدعم عمليات بايثون المدمجة:

إنشاء وتدمير الكائنات

class Resource: def __new__(cls, *args, **kwargs): # يُستدعى أولاً لإنشاء كائن جديد (نادراً ما يُستخدم) print('Creating instance') instance = super().__new__(cls) return instance def __init__(self, name): # يُستدعى بعد __new__ لتهيئة الكائن self.name = name print(f'Initializing {name}') def __del__(self): # يُستدعى عند حذف الكائن (destructor) print(f'Destroying {self.name}')
🔍 __new__ و __init__ و __del__ - دورة حياة الكائن

__new__ هو أول ميثود يُستدعى عند إنشاء كائن، وهو المسؤول عن تخصيص الذاكرة (memory allocation) وإرجاع الكائن الجديد. يُستخدم __new__ بشكل أساسي مع الأنماط غير القابلة للتغيير (immutable) مثل int و str أو مع Singleton pattern. يليه مباشرة __init__ الذي يقوم بتهيئة الكائن (initialization) وتعيين القيم الأولية للخصائص. __del__ هو الـ destructor ويُستدعى عندما يتم حذف الكائن (عندما يصل reference count إلى الصفر أو عند تشغيل garbage collector). من المهم ملاحظة أن __del__ لا يُستدعى بالضرورة فور استخدام del، وقد لا يُستدعى أبداً.

تمثيل الكائن كنص

class Book: def __init__(self, title, author, pages): self.title = title self.author = author self.pages = pages def __str__(self): # للمستخدم النهائي (print, str()) return f"'{self.title}' by {self.author}" def __repr__(self): # للمطور (debugging, repr()) - يجب أن يكون واضحاً return f"Book('{self.title}', '{self.author}', {self.pages})" def __len__(self): return self.pages book = Book('Python Crash Course', 'Eric Matthes', 560) print(book) # 'Python Crash Course' by Eric Matthes print(repr(book)) # Book('Python Crash Course', 'Eric Matthes', 560) print(len(book)) # 560
🔍 __str__ و __repr__ و __len__ - تمثيل الكائنات

__str__ يُستخدم لإنشاء تمثيل readable للمستخدم النهائي ويُستدعى عند استخدام print() أو str(). __repr__ يُستخدم لإنشاء تمثيل unambiguous للمطور ويجب أن يكون واضحاً كافياً لإعادة إنشاء الكائن (ideal: eval(repr(obj)) == obj). __len__ يسمح باستخدام دالة len() المدمجة مع الكائن، وتستخدم عادة مع containers. الفرق الجوهري: __str__ للقراءة البشرية، __repr__ للـ debugging وإعادة الإنشاء. إذا لم تعرّف __str__ فستستخدم بايثون __repr__ كبديل.

المقارنة والتجزئة

class Person: def __init__(self, name, age): self.name = name self.age = age def __eq__(self, other): # == if not isinstance(other, Person): return NotImplemented return self.name == other.name and self.age == other.age def __lt__(self, other): # < return self.age < other.age def __hash__(self): # لجعل الكائن hashable (للاستخدام في set, dict key) return hash((self.name, self.age)) # @total_ordering - يملأ طرق المقارنة الناقصة تلقائياً from functools import total_ordering @total_ordering class Employee: def __init__(self, name, salary): self.name = name self.salary = salary def __eq__(self, other): return self.salary == other.salary def __lt__(self, other): return self.salary < other.salary # الآن __le__, __gt__, __ge__ متوفرة تلقائياً!
🔍 __eq__ و __hash__ و @total_ordering

__eq__ يعرّف سلوك عامل المساواة == ويجب أن يعيد NotImplemented إذا كان الكائن الآخر من نوع غير متوافق (بدلاً من رفع TypeError). __lt__ يعرّف < ويستخدم في عمليات الترتيب. __hash__ يجعل الكائن hashable للاستخدام كـ key في القواميس أو عنصر في المجموعات (set). قاعدة مهمة: إذا عرّفت __eq__ يجب عادة تعريف __hash__ أيضاً (وإلا سيصبح الكائن unhashable). الديكوريتور @total_ordering من functools يوفّر الوقت: يعرّف __eq__ و __lt__ فقط ويولّد تلقائياً __le__ و __gt__ و __ge__ بناءً عليهما.

العمليات الحسابية

class Vector: def __init__(self, x, y): self.x = x; self.y = y def __add__(self, other): # + return Vector(self.x + other.x, self.y + other.y) def __sub__(self, other): # - return Vector(self.x - other.x, self.y - other.y) def __mul__(self, scalar): # * return Vector(self.x * scalar, self.y * scalar) def __rmul__(self, scalar): # 3 * v (عندما يفشل __mul__ على اليسار) return self.__mul__(scalar) def __neg__(self): # -v return Vector(-self.x, -self.y) def __abs__(self): # abs(v) return (self.x**2 + self.y**2)**0.5 def __repr__(self): return f'Vector({self.x}, {self.y})' v1 = Vector(2, 4) v2 = Vector(1, 3) print(v1 + v2) # Vector(3, 7) print(v1 * 3) # Vector(6, 12) print(3 * v1) # Vector(6, 12) - thanks to __rmul__ print(abs(v1)) # 4.472...
🔍 __add__ و __rmul__ والعمليات الحسابية المخصصة

هذا المثال يوضح كيفية جعل الكلاس يدعم العمليات الحسابية المدمجة. __add__ يعرّف الجمع (+)، __sub__ للطرح، __mul__ للضرب، __neg__ للسالب الأحادي (-v)، و __abs__ للقيمة المطلقة. المهم هنا هو __rmul__ (reflected multiplication): عندما تكتب 3 * v1 وتحاول بايثون استدعاء (3).__mul__(v1) ويفشل، تبحث عن v1.__rmul__(3). هذا يسمح للعملية بالعمل في كلا الاتجاهين. كل عملية حسابية لها reflected counterpart (__radd__, __rsub__, إلخ) تُستدعى عندما يكون المعامل الأيسر لا يدعم العملية.

الوصول إلى العناصر كـ Container

class Playlist: def __init__(self): self.songs = [] def __getitem__(self, index): # playlist[0] return self.songs[index] def __setitem__(self, index, value): # playlist[0] = song self.songs[index] = value def __delitem__(self, index): # del playlist[0] del self.songs[index] def __contains__(self, item): # 'song' in playlist return item in self.songs def __len__(self): # len(playlist) return len(self.songs) def __iter__(self): # for song in playlist return iter(self.songs) def __reversed__(self): # reversed(playlist) return reversed(self.songs)
🔍 Container Protocols - محاكاة القوائم والقواميس

هذه الميثودز تحوّل أي كلاس إلى container كامل الوظائف يشبه القوائم. __getitem__ يسمح بالوصول عبر الفهرس (playlist[0])، __setitem__ يسمح بتعيين القيم، و __delitem__ يسمح بالحذف. __contains__ يفعّل استخدام in للتحقق من وجود عنصر. __iter__ يجعل الكائن قابلاً للتكرار في حلقات for. __reversed__ يسمح باستخدام reversed(). بتطبيق هذه الميثودز، يصبح كلاسك متوافقاً تماماً مع واجهة الـ containers البايثونية القياسية ويمكن استخدامه مع دوال مثل sorted() و enumerate().

Context Managers - __enter__ و __exit__

class ManagedFile: def __init__(self, filename, mode): self.filename = filename self.mode = mode def __enter__(self): self.file = open(self.filename, self.mode) return self.file # القيمة التي يأخذها 'as' def __exit__(self, exc_type, exc_val, exc_tb): self.file.close() if exc_type: print(f'An error occurred: {exc_val}') return True # يمنع انتشار الاستثناء with ManagedFile('test.txt', 'w') as f: f.write('Hello, world!')
🔍 Context Managers - __enter__ و __exit__

Context Managers تدير الموارد عبر two-phase protocol: __enter__ يُستدعى عند الدخول إلى كتلة with ويعيد القيمة المرتبطة بـ as (مثل الملف المفتوح). __exit__ يُستدعى عند الخروج من الكتلة (سواء بنجاح أو بخطأ) ويستقبل three arguments: نوع الاستثناء وقيمته و traceback. إذا أعاد __exit__ True، فسيتم منع انتشار الاستثناء (أي التعامل معه ضمنياً). هذا النمط يضمن تحرير الموارد حتى في حالة حدوث أخطاء، وهو أفضل من try/finally التقليدي. الاستخدام الأكثر شيوعاً هو with open() as f الذي يغلق الملف تلقائياً.

Callable Objects - __call__

class Counter: def __init__(self): self.count = 0 def __call__(self, *args, **kwargs): self.count += 1 print(f'Called {self.count} times') return self.count counter = Counter() counter() # Called 1 times counter() # Called 2 times counter() # Called 3 times print(callable(counter)) # True
🔍 __call__ - جعل الكائنات قابلة للاستدعاء

باستخدام __call__ يمكن جعل أي كائن قابلاً للاستدعاء مثل الدوال (counter()). هذا مفيد جداً لإنشاء كائنات تحتفظ بحالة بين الاستدعاءات (stateful callables). مثلاً، كلاس Counter يتتبع عدد المرات التي استُدعي فيها. دالة callable() تتحقق مما إذا كان الكائن قابلاً للاستدعاء. __call__ يُستخدم في العديد من التطبيقات مثل decorators (التي تحتاج لحفظ حالة) واستراتيجيات الـ partial application والـ callback objects المتقدمة. هذا النمط يسمى "closures مع حالة" أو "functor".

6. Decorators (الديكوريتورز) - دليل كامل

Decorators هي دوال تعدل سلوك دوال أخرى. هي أداة قوية جداً في بايثون:

مبدأ العمل - Functions كـ First-Class Citizens

# الدوال في بايثون هي كائنات من الدرجة الأولى def greet(name): return f'Hello {name}' # يمكننا تخزين دالة في متغير my_func = greet print(my_func('Ahmed')) # Hello Ahmed # يمكننا تمرير دالة كوسيط def call_func(func, arg): return func(arg) print(call_func(greet, 'Sara')) # Hello Sara # يمكننا تعريف دالة داخل دالة (Nested function) def outer(message): def inner(): print(message) return inner # نعيد الدالة الداخلية (Closure) closure = outer('Hello from closure!') closure() # Hello from closure!
🔍 First-Class Functions و Closures

في بايثون، الدوال هي كائنات من الدرجة الأولى (first-class citizens): يمكن تخزينها في متغيرات، تمريرها كوسيط لدوال أخرى، وإعادتها كقيمة من دوال أخرى. المثال يوضح كل هذه المفاهيم: my_func = greet تخزين دالة، call_func(greet, arg) تمرير دالة كوسيط، و outer تعيد الدالة الداخلية inner. هذا الأخير يسمى Closure: دالة داخلية تتذكر البيئة التي أنشئت فيها (المتغير message) حتى بعد انتهاء الدالة الخارجية. Closures هي الأساس الذي تبنى عليه الـ decorators.

إنشاء Decorator بسيط

def my_decorator(func): def wrapper(*args, **kwargs): print('Before function call') result = func(*args, **kwargs) print('After function call') return result return wrapper @my_decorator def say_hello(name): print(f'Hello {name}') say_hello('Ahmed') # Before function call # Hello Ahmed # After function call
🔍 Decorator بسيط - إضافة سلوك قبل وبعد الدالة

الـ decorator هو دالة تستقبل دالة أخرى وتعيد دالة جديدة تحل محلها. @my_decorator هو syntactic sugar للكتابة say_hello = my_decorator(say_hello). الدالة wrapper تستخدم *args و **kwargs لتكون مرنة وتقبل أي وسائط. تقوم بتنفيذ كود قبل وبعد استدعاء الدالة الأصلية، وتعيد نتيجتها. هذا النمط يستخدم في الـ logging والـ timing والتحقق من الصلاحيات والتخزين المؤقت (caching). الـ decorators هي واحدة من أقوى ميزات بايثون وتستخدم على نطاق واسع في أطر العمل مثل Flask و Django.

Decorators مع وسائط

def repeat(num_times): def decorator(func): def wrapper(*args, **kwargs): for _ in range(num_times): result = func(*args, **kwargs) return result return wrapper return decorator @repeat(num_times=3) def greet(name): print(f'Hello {name}') greet('World') # Hello World (تتكرر 3 مرات)
🔍 Decorators مع وسائط - multilayer wrappers

عندما يحتاج الـ decorator لوسائط خاصة به (num_times)، نحتاج لثلاث مستويات من الدوال المتداخلة: repeat تأخذ الوسائط وتعيد decorator الذي يأخذ الدالة الأصلية ويعيد wrapper. هذا يكافئ greet = repeat(num_times=3)(greet). هذا النمط شائع في مكتبات مثل Flask (@app.route('/path')) و @click.option('--name'). يمكنك التفكير فيها كـ "مصنع decorators" (decorator factory) ينتج decorator مخصص حسب الوسائط المعطاة.

الحفاظ على Metadata باستخدام functools.wraps

from functools import wraps def my_decorator(func): @wraps(func) # يحافظ على اسم الدالة و docstring def wrapper(*args, **kwargs): """Wrapper function""" print('Before') return func(*args, **kwargs) return wrapper @my_decorator def say_hello(): """Say hello function""" print('Hello!') print(say_hello.__name__) # say_hello (بدون wraps: wrapper) print(say_hello.__doc__) # Say hello function
🔍 functools.wraps - الحفاظ على هوية الدالة

بدون @wraps(func)، ستفقد الدالة المزخرفة هويتها: يصبح __name__ يساوي 'wrapper' و __doc__ يساوي 'Wrapper function' بدلاً من القيم الأصلية. @wraps ينسخ metadata الدالة الأصلية (__name__، __doc__، __module__، __annotations__، وغيرها) إلى الدالة wrapper. هذا أمر بالغ الأهمية لـ debugging والتوثيق التلقائي. دائماً استخدم @wraps عند كتابة أي decorator - إنها ممارسة قياسية لا تهملها.

Decorator لقياس وقت التنفيذ

import time from functools import wraps def timer(func): @wraps(func) def wrapper(*args, **kwargs): start = time.perf_counter() result = func(*args, **kwargs) end = time.perf_counter() print(f'{func.__name__} took {end - start:.6f}s') return result return wrapper @timer def slow_function(): time.sleep(1) return 'Done' slow_function() # slow_function took 1.000123s
🔍 @timer Decorator - قياس وقت التنفيذ

هذا مثال عملي على decorator يقيس وقت تنفيذ أي دالة باستخدام time.perf_counter() (أعلى دقة متاحة). يطبع اسم الدالة ووقت التنفيذ بالثواني. هذا النوع من الـ decorators مفيد جداً في تحليل أداء الكود (profiling) واكتشاف bottlenecks. يمكن توسيعه ليشمل عدد الاستدعاءات أو تنسيقات مختلفة للوقت. باستخدام @wraps(func) نضمن أن الدوال المزخرفة تحتفظ بأسمائها الأصلية. يمكنك تطبيق @timer على أي دالة لقياس أدائها بدون تعديل كود الدالة نفسها.

Class Decorators

def singleton(cls): """Decorator لتحويل كلاس إلى Singleton""" instances = {} @wraps(cls) def get_instance(*args, **kwargs): if cls not in instances: instances[cls] = cls(*args, **kwargs) return instances[cls] return get_instance @singleton class Database: def __init__(self): print('Connecting to database...') db1 = Database() # Connecting to database... db2 = Database() # لا يطبع شيئاً - نفس النسخة! print(db1 is db2) # True
🔍 Class Decorator - نمط Singleton

الـ class decorator يعمل على الكلاس نفسه بدلاً من الدوال. هنا @singleton يحوّل كلاس Database إلى singleton: مهما حاولت إنشاء نسخ متعددة ستحصل دائماً على نفس النسخة. الآلية بسيطة: قاموس instances يخزن النسخة الأولى لكل كلاس، وأي استدعاء لاحق يعيد نفس النسخة. لاحظ أن استخدام @wraps(cls) يحافظ على metadata الكلاس. هذه طريقة أنيقة لتطبيق singleton مقارنة بالطرق الأخرى مثل metaclasses. من المهم تنبيه أن class decorator يعيد دالة (get_instance) وليس كلاساً، لكن هذا الشفاف يعمل في معظم الحالات.

Decorator متقدم - التحقق من الصلاحيات

from functools import wraps def requires_auth(roles=None): def decorator(func): @wraps(func) def wrapper(user, *args, **kwargs): if not user.get('authenticated'): raise PermissionError('User not authenticated') if roles and user.get('role') not in roles: raise PermissionError('Insufficient permissions') return func(user, *args, **kwargs) return wrapper return decorator @requires_auth(roles=['admin']) def delete_user(user, user_id): print(f'User {user_id} deleted') admin = {'authenticated': True, 'role': 'admin'} delete_user(admin, 1) # يعمل بنجاح
🔍 @requires_auth - التحقق من الصلاحيات بالديكوريتورز

هذا decorator متقدم يتحقق من صلاحيات المستخدم قبل تنفيذ الدالة. يستخدم نمط decorator مع وسائط (ثلاثة مستويات) ويقبل قائمة بالأدوار المسموح لها. يقوم أولاً بالتحقق من أن المستخدم مصادق (authenticated) ثم يتأكد أن دوره ضمن الأدوار المسموحة. إذا فشل أي شرط يرفع PermissionError. هذا النمط شائع جداً في أطر عمل الويب مثل Flask و Django حيث تُستخدم decorators لحماية endpoints. الميزة الأساسية: فصل منطق التحقق من الصلاحيات عن منطق العمل الأساسي (separation of concerns).

7. Descriptors - التحكم في الوصول للخصائص

Descriptors تسمح بتخصيص سلوك الوصول إلى attributes:

class PositiveNumber: def __set_name__(self, owner, name): self.name = f'_{name}' def __get__(self, instance, owner): return getattr(instance, self.name) def __set__(self, instance, value): if value <= 0: raise ValueError('Value must be positive') setattr(instance, self.name, value) class Order: quantity = PositiveNumber() price = PositiveNumber() def __init__(self, quantity, price): self.quantity = quantity self.price = price order = Order(5, 10.0) # order.quantity = -1 # ValueError!
🔍 Descriptors - إعادة استخدام منطق التحقق من القيم

الـ Descriptor Protocol (__get__ و __set__ و __set_name__) يسمح بإنشاء كائنات تتحكم في الوصول إلى خصائص الكلاسات الأخرى. PositiveNumber هو data descriptor يضمن أن القيمة موجبة دائماً. __set_name__ يُستدعى تلقائياً عند تعريف الكلاس لتحديد اسم الخاصية. هذا النمط أفضل من تكرار كود التحقق في كل @property لأنه يعيد استخدام منطق التحقق عبر خصائص متعددة. بايثون نفسها تستخدم descriptors داخلياً في @property و @classmethod و @staticmethod. الفرق بين data descriptor (يملك __set__) و non-data descriptor يحدد أولوية الوصول.

8. __slots__ - تحسين الأداء والذاكرة

يمنع إنشاء __dict__ ويوفر ذاكرة للكلاسات التي تحتوي آلاف النسخ:

class Point: __slots__ = ('x', 'y') def __init__(self, x, y): self.x = x self.y = y # لا يمكن إضافة خصائص غير معرفة في __slots__ # p.z = 5 # AttributeError!
🔍 __slots__ - توفير الذاكرة ومنع الخصائص الديناميكية

__slots__ يمنع إنشاء __dict__ لكل كائن ويخزن الخصائص في array ذي حجم ثابت، مما يقلل استهلاك الذاكرة بشكل كبير (يصل إلى 40-50% توفيراً). كل كائن بايثون يحمل __dict__ وهو قاموس يستهلك ذاكرة إضافية (hashing overhead). باستخدام __slots__، نحدد الخصائص المسموح بها مسبقاً ولا يمكن إضافة خصائص جديدة ديناميكياً (محاولة تعيين p.z = 5 سترفع AttributeError). هذا مفيد جداً في الكلاسات التي تنشئ آلاف أو ملايين النسخ مثل تمثيل البيانات في machine learning أو معالجة datasets ضخمة. مع ذلك، __slots__ له تكلفة: لا يمكن استخدامه مع الوراثة بسهولة (يجب تعريفه في كل كلاس ابن)، وقد تتعطل بعض الأدوات التي تعتمد على __dict__.

9. مبادئ SOLID في OOP - مع أمثلة عملية

SOLID هي خمس مبادئ أساسية لكتابة كود كائني قابل للصيانة والتوسع. إليك كل مبدأ مع مثال عملي:

S - Single Responsibility Principle (SRP)

كل Class يجب أن يكون له سبب واحد فقط للتغيير (مسؤولية واحدة):

# قبل - مخالف لـ SRP: كلاس واحد يفعل كل شيء class Report: def __init__(self, data): self.data = data def generate(self): ... def save_to_db(self): ... # مسئولية زائدة def send_email(self): ... # مسئولية زائدة # بعد - كل class له مسئولية واحدة class ReportGenerator: def generate(self, data): ... class ReportRepository: def save(self, report): ... class EmailService: def send(self, report, recipient): ...
🔍 S - مبدأ المسؤولية الواحدة (SRP)

يقول المبدأ: "يجب أن يكون لكل كلاس سبب واحد فقط للتغيير". قبل التطبيق، كلاس Report يتحمل مسؤوليات متعددة: إنشاء التقرير، حفظه في قاعدة البيانات، وإرساله عبر البريد. هذا يعني أن أي تغيير في سياسة الحفظ أو الإرسال سيؤدي لتعديل نفس الكلاس. بعد التطبيق، فصلنا المسؤوليات إلى ثلاث كلاسات متخصصة: ReportGenerator للإنشاء، ReportRepository للحفظ، و EmailService للإرسال. فوائد هذا الفصل: سهولة الاختبار (يمكن اختبار كل كلاس بمفرده)، سهولة التعديل (تغيير طريقة الحفظ لا يؤثر على منطق الإنشاء)، وإعادة الاستخدام (يمكن استخدام EmailService مع أنواع تقارير أخرى).

O - Open/Closed Principle (OCP)

مفتوح للامتداد (extension) ومغلق للتعديل (modification). استخدم الوراثة والتركيب بدلاً من تغيير الكود الموجود:

# قبل - تعديل الكود كلما أضفنا shape جديد class AreaCalculator: def calculate(self, shapes): total = 0 for shape in shapes: if isinstance(shape, Circle): total += 3.14 * shape.radius ** 2 elif isinstance(shape, Square): total += shape.side ** 2 return total # بعد - مفتوح للامتداد عبر abstractmethod from abc import ABC, abstractmethod class Shape(ABC): @abstractmethod def area(self): pass class Circle(Shape): def __init__(self, radius): self.radius = radius def area(self): return 3.14 * self.radius ** 2 class Square(Shape): def __init__(self, side): self.side = side def area(self): return self.side ** 2 # AreaCalculator لم يتغير! أي Shape جديد يضاف بدون تعديله class AreaCalculator: def calculate(self, shapes): return sum(shape.area() for shape in shapes)
🔍 O - مبدأ الـ Open/Closed (OCP)

مبدأ OCP يقول: "مفتوح للامتداد، مغلق للتعديل". قبل التطبيق، كلما أضفنا شكلاً هندسياً جديداً، نضطر لتعديل AreaCalculator وإضافة elif جديد - هذا يخالف OCP. بعد التطبيق، استخدمنا Shape كـ abstract base class مع @abstractmethod area(). أي شكل جديد يرث من Shape ويطبق area() بدون أي تعديل على AreaCalculator. الامتداد يحدث عبر إنشاء كلاسات جديدة، وليس بتعديل كلاس موجود. هذا هو power of polymorphism: calculate يستخدم shape.area() ولا يهتم بنوع الشكل.

L - Liskov Substitution Principle (LSP)

الكائنات المشتقة يجب أن تحل محل الكائنات الأساسية دون تغيير في صحة البرنامج. المربع ليس مستطيلاً في OOP:

# قبل - Rectangle له setter منفصل لكل بُعد، Square يكسره! class Rectangle: def __init__(self, w, h): self._w, self._h = w, h def set_width(self, w): self._w = w def set_height(self, h): self._h = h def area(self): return self._w * self._h class Square(Rectangle): # LSP Violation! def set_width(self, w): self._w = self._h = w def set_height(self, h): self._w = self._h = h # استخدام: الكود يفترض أن تغيير width لا يؤثر على height def test_area(rect): rect.set_width(5) rect.set_height(4) assert rect.area() == 20 # Square: 4*4=16! فشل! # بعد - استخدام Abstract Class مشترك class Shape(ABC): @abstractmethod def area(self): pass class Rectangle(Shape): def __init__(self, w, h): self.w, self.h = w, h def area(self): return self.w * self.h class Square(Shape): def __init__(self, side): self.side = side def area(self): return self.side ** 2
🔍 Abstract Base Classes (ABC) وفرض الواجهات

ABCs تسمح لك بتعريف واجهات (interfaces) تجبر الكلاسات المشتقة على تنفيذ ميثودز محددة. الميثودز المزينة بـ @abstractmethod يجب أن تُنفذ في أي كلاس غير abstract يرث من Shape. إذا حاولت إنشاء كائن من Shape مباشرة، ستحصل على TypeError. يمكن أيضاً استخدام @abstractmethod مع @property كما في description لفرض تنفيذ خصائص معينة. هذا يضمن أن كل الأشكال الهندسية ستوفر واجهة موحدة (area, perimeter, description) مع تنفيذ مختلف لكل منها.

I - Interface Segregation Principle (ISP)

الواجهات المتخصصة أفضل من واجهة عامة واحدة. لا تجبر الكلاسات على تنفيذ methods لا تحتاجها:

# قبل - واجهة عامة تفرض methods غير مرغوب فيها class Worker(ABC): @abstractmethod def work(self): pass @abstractmethod def eat(self): pass # Robot لا يأكل! class Robot(Worker): # مجبر على تنفيذ eat! def work(self): ... def eat(self): raise NotImplementedError('Robots do not eat') # بعد - واجهات متخصصة class Workable(ABC): @abstractmethod def work(self): pass class Eatable(ABC): @abstractmethod def eat(self): pass class Human(Workable, Eatable): ... class Robot(Workable): ...
🔍 I - مبدأ فصل الواجهات (ISP)

مبدأ ISP يقول: "لا تجبر العميل على الاعتماد على واجهات لا يستخدمها". قبل التطبيق، Worker يحتوي على eat و work. كلاس Robot يضطر لتنفيذ eat() رغم أن الروبوتات لا تأكل، مما يسبب NotImplementedError. بعد التطبيق، فصلنا الواجهات إلى Workable و Eatable. Robot يطبق فقط Workable، بينما Human يطبق كليهما. هذا يجعل الكود أكثر مرونة ويتبع مبدأ "مسؤولية واحدة" على مستوى الواجهات.

D - Dependency Inversion Principle (DIP)

اعتمد على التجريد (Abstract) لا على التنفيذ (Concrete). الكلاسات عالية المستوى لا تعتمد على كلاسات منخفضة المستوى:

# قبل - اعتماد مباشر على MySQLDatabase class UserService: def __init__(self): self.db = MySQLDatabase() # اقتران قوي! # بعد - الاعتماد على التجريد (Dependency Injection) class Database(ABC): @abstractmethod def save(self, data): pass @abstractmethod def find(self, id): pass class PostgreSQLDatabase(Database): def save(self, data): print('PostgreSQL: saving...') def find(self, id): ... class MongoDatabase(Database): def save(self, data): print('MongoDB: saving...') def find(self, id): ... class UserService: def __init__(self, db: Database): # يعتمد على التجريد! self.db = db # يمكن تبديل قاعدة البيانات دون تغيير UserService! service = UserService(PostgreSQLDatabase()) service2 = UserService(MongoDatabase())
🔍 D - مبدأ عكس التبعيات (DIP)

مبدأ DIP يقول: "اعتمد على التجريدات، وليس على التنفيذات الملموسة". قبل التطبيق، UserService يعتمد مباشرة على MySQLDatabase (اقتران قوي). لتغيير قاعدة البيانات، يجب تعديل UserService. بعد التطبيق، عرّفنا Database كـ ABC وأصبح UserService يعتمد على التجريد. يمكننا الآن تمرير أي نوع قاعدة بيانات (PostgreSQLDatabase, MongoDatabase) بدون تغيير UserService. هذا هو Dependency Injection: حقن التبعيات من الخارج يقلل الاقتران ويسهل الاختبار.

10. Metaclasses - كلاسات تصنع الكلاسات

Metaclass هو "كلاس الكلاس". تماماً كما أن الكلاس يصنع الكائنات، الـ metaclass يصنع الكلاسات. في بايثون، النوع الافتراضي لـ metaclass هو type:

# type() كـ metaclass MyClass = type('MyClass', (BaseClass,), {'attr': 1}) # يساوي تماماً: class MyClass(BaseClass): attr = 1
🔍 Metaclasses - type() كمنشئ للكلاسات

في بايثون، الكلاسات نفسها هي كائنات من نوع type. العبارة type('MyClass', (BaseClass,), {'attr': 1}) تنشئ كلاساً جديداً ديناميكياً بنفس الطريقة التي يعمل بها تعريف class. هذا يعني أن type هو metaclass افتراضي لجميع الكلاسات. metaclass هو "كلاس الكلاس": يتحكم في كيفية إنشاء الكلاسات وتصرفها. فهم هذا المفهوم يفتح الباب لتقنيات متقدمة مثل إنشاء APIs ديناميكية (مثل SQLAlchemy و Django ORM) وتعديل سلوك الكلاسات بشكل آلي.

متى تحتاج Metaclass؟

  • تريد تعديل سلوك إنشاء الكلاس نفسه (وليس الكائنات)
  • تريد فرض واجهات أو قواعد في كل الكلاسات في نظامك
  • تريد تسجيل/توثيق كل الكلاسات التي تنشأ (مثل ORM: SQLAlchemy, Django)
  • تريد تحويل attributes تلقائياً (مثل dataclasses)
class SingletonMeta(type): _instances = {} def __call__(cls, *args, **kwargs): if cls not in cls._instances: cls._instances[cls] = super().__call__(*args, **kwargs) return cls._instances[cls] class Database(metaclass=SingletonMeta): def __init__(self): print('Connection established') db1 = Database() # Connection established db2 = Database() # لا يطبع - نفس النسخة! print(db1 is db2) # True
🔍 Metaclass Singleton - التحكم في إنشاء النسخ

هذا metaclass يطبق نمط Singleton عبر override __call__ في الـ metaclass. الفرق عن class decorator أن metaclass يحافظ على نوع الكائن (type identity). __call__ في metaclass يُستدعى عند إنشاء كائن من الكلاس (Database()). يتحقق أولاً إذا كان هناك نسخة موجودة، وإلا ينشئ واحدة عبر super().__call__(). هذا يضمن نسخة واحدة لكل كلاس يستخدم هذا metaclass. هذه الطريقة هي الأفضل للـ singleton عندما تحتاج الوراثة أو الحفاظ على typing صحيح.

# Metaclass لإضافة prefix تلقائي لطرق الكلاس class PrefixMeta(type): def __new__(cls, name, bases, namespace): # namespace هو dict لكل attributes الكلاس prefix = namespace.get('prefix', '') new_namespace = {} for key, value in namespace.items(): if callable(value) and not key.startswith('__'): new_namespace[prefix + key] = value else: new_namespace[key] = value return super().__new__(cls, name, bases, new_namespace) class APIEndpoints(metaclass=PrefixMeta): prefix = 'api_' def get_user(self): pass def create_post(self): pass api = APIEndpoints() api.api_get_user() # method name changed! api.api_create_post() # method name changed!
🔍 Metaclass لتعديل Namespace

هذا metaclass يعدّل محتوى الكلاس عند إنشائه بإضافة prefix (api_) لأسماء الميثودز. __new__ في metaclass يستقبل namespace (قاموس خصائص الكلاس) قبل إنشاء الكلاس. نمر عليه ونضيف البادئة لكل ميثود غير خاص (لا يبدأ بـ __). هذا النوع من المعالجة التلقائية هو أساس أطر العمل مثل Django و SQLAlchemy حيث يتم تحويل تعريفات الكلاسات إلى جداول قاعدة بيانات أو نماذج. metaclasses تمنحك القدرة على "برمجة" كيفية تعريف الكلاسات نفسها.

11. Descriptors - تحكم كامل في الخصائص

Descriptor هو أي كائن يطبق أحد هذه الميثودز: __get__, __set__, __delete__. بايثون تستخدمهم في @property, @classmethod, @staticmethod.

Data vs Non-Data Descriptors

# Non-data descriptor (فقط __get__) - أقل أولوية من instance dict class NonDataDescriptor: def __get__(self, instance, owner): return 'from descriptor' # Data descriptor (__get__ + __set__) - أولوية أعلى من instance dict class DataDescriptor: def __get__(self, instance, owner): return getattr(instance, '_val', 'default') def __set__(self, instance, value): setattr(instance, '_val', value) class Test: nd = NonDataDescriptor() d = DataDescriptor() t = Test() t.nd = 'instance value' t.d = 'instance value' print(t.nd) # 'instance value' (instance dict يفوق non-data desc) print(t.d) # 'default' (data desc يفوق instance dict!)
🔍 Data vs Non-Data Descriptors

الفرق بين data descriptor (يملك __set__ أو __delete__) و non-data descriptor (يملك __get__ فقط) يحدد أولوية الوصول. Data descriptor له أولوية أعلى من __dict__ الخاص بالكائن (instance dict). هذا يفسر لماذا t.d = 'instance value' لا يتجاوز DataDescriptor (لا يزال t.d يعيد القيمة من descriptor). أما non-data descriptor فأولوته أقل من instance dict، لذلك t.nd = 'instance value' يتجاوزه. هذا التمييز مهم جداً في فهم كيفية عمل @property و @classmethod و @staticmethod.

property() باستخدام Descriptor Protocol

# تنفيذ property() من الصفر باستخدام descriptors class Property: def __init__(self, fget=None, fset=None, fdel=None): self.fget = fget; self.fset = fset; self.fdel = fdel def __get__(self, instance, owner): if instance is None: return self return self.fget(instance) def __set__(self, instance, value): if self.fset is None: raise AttributeError("can't set attribute") self.fset(instance, value) def setter(self, fset): return Property(self.fget, fset, self.fdel) class Person: def __init__(self, name): self._name = name @Property def name(self): return self._name @name.setter def name(self, value): if not value: raise ValueError('Name required') self._name = value
🔍 تنفيذ @property من الصفر باستخدام Descriptors

هذا مثال متقدم يوضح كيفية بناء آلية @property بنفسك باستخدام descriptor protocol. كلاس Property يخزن الدوال fget، fset، fdel التي تمثل getter و setter و deleter. __get__ يستدعي self.fget(instance)، __set__ يستدعي self.fset(instance, value) مع التحقق من وجوده. الميثود setter يعيد كائن Property جديداً مع fset المحدد. هذا يشرح كيف يعمل @property داخلياً: هو مجرد data descriptor يوفر sugary syntax للـ getter/setter pattern.

12. Context Managers - إدارة الموارد باحترافية

Context Managers تدير الموارد (ملفات، اتصالات قاعدة بيانات، locks) وتضمن تحريرها حتى مع حدوث أخطاء:

طريقتان للتنفيذ

# الطريقة 1: كلاس مع __enter__ و __exit__ class DatabaseConnection: def __init__(self, dsn): self.dsn = dsn def __enter__(self): self.conn = connect(self.dsn) return self.conn def __exit__(self, exc_type, exc_val, exc_tb): self.conn.close() if exc_type: # حدث خطأ print(f'Error: {exc_val}') return False # لا تمنع انتشار الخطأ (True تمنعه) # الطريقة 2: contextmanager decorator مع generator from contextlib import contextmanager @contextmanager def database_connection(dsn): print('Connecting...') conn = connect(dsn) try: yield conn # ما قبل yield = __enter__, ما بعد = __exit__ finally: conn.close() print('Connection closed') # الاستخدام (نفس الشيء للطريقتين) with DatabaseConnection('postgresql://...') as conn: conn.execute('SELECT * FROM users')
🔍 Context Managers - طريقتان للتنفيذ

(1) عبر كلاس مع __enter__ و __exit__: مناسب لـ context managers معقدة تحتاج حالة (state). (2) عبر @contextmanager مع generator: أبسط وأنسب للحالات البسيطة. في الطريقة الأولى، نتحكم بالدخول والخروج بشكل صريح. في الطريقة الثانية، yield يفصل بين ما ينفذ عند الدخول (قبل yield) وعند الخروج (بعد yield). كتلة finally تضمن التنظيف حتى مع الاستثناءات. كلا الطريقتين متكافئتان، اختر حسب التعقيد.

contextlib.nested و ExitStack

from contextlib import ExitStack # إدارة موارد متعددة ديناميكياً def process_files(filenames): with ExitStack() as stack: files = [stack.enter_context(open(f)) for f in filenames] # كل الملفات ستُغلق تلقائياً عند الخروج من with return [f.read() for f in files]
🔍 ExitStack - إدارة موارد متعددة ديناميكياً

ExitStack يسمح بإدارة عدد متغير من الموارد داخل with. كل مورد يُضاف عبر stack.enter_context() سيتم تنظيفه تلقائياً عند الخروج من with بترتيب عكسي (LIFO). هذا مفيد جداً عندما لا تعرف عدد الموارد مسبقاً (مثل قائمة ملفات). ExitStack يتعامل مع الاستثناءات بشكل صحيح: إذا فشل فتح ملف، سيتم إغلاق الملفات التي فتحت قبله. هذه أداة متقدمة من contextlib يجب أن يعرفها كل مطور بايثون محترف.

نصيحة أداء: استخدم contextlib.contextmanager للـ context managers البسيطة، واستخدم class للـ complex ones التي تحتاج حالة معقدة.

13. Composition - التركيب (Has-A) بدل الوراثة (Is-A)

الـ Composition يعني "الكائن يحتوي على كائن آخر" بدلاً من "الكائن هو نوع من كائن آخر". هذه قاعدة ذهبية: "فضّل التركيب على الوراثة" (Favor Composition over Inheritance):

# قبل - وراثة (Is-A): مرتبطة بقوة، يصعب تغييرها class Car: def start(self): return 'Starting...' class ElectricCar(Car): # Is-A Car - لكن هل هذا دقيق؟ def start(self): return 'Starting silently...' class FlyingCar(Car): # Is-A Car + can fly def start(self): return 'Starting with wings...' # بعد - Composition (Has-A): مرن، كل جزء مستقل class Engine: def start(self): return 'Engine started' class ElectricMotor: def start(self): return 'Motor running silently' class Wings: def deploy(self): return 'Wings deployed' class Car: def __init__(self, engine: Engine): # Has-A Engine self.engine = engine # نركب المكونات حسب الحاجة - بدون كلاسات كثيرة! gas_car = Car(Engine()) electric_car = Car(ElectricMotor()) flying_car = Car(Engine()) flying_car.wings = Wings() # نضيف مكون إضافي print(gas_car.engine.start()) # Engine started print(electric_car.engine.start()) # Motor running silently
🔍 Composition (Has-A) - التركيب بدل الوراثة

الـ Composition يعني "الكائن يحتوي على كائن آخر" بدلاً من "الكائن هو نوع من كائن آخر". كلاس Car يحتوي على Engine (أو ElectricMotor) بدلاً من أن يكون نوعاً خاصاً من السيارة لكل محرك. الميزة: يمكن تركيب المكونات حسب الحاجة. تريد سيارة كهربائية؟ استخدم Car(ElectricMotor()). تريد إضافة أجنحة؟ flying_car.wings = Wings(). هذا أكثر مرونة من هرم وراثي معقد. القاعدة: "فضّل التركيب على الوراثة" (Favor Composition over Inheritance).

Dependency Injection - حقن التبعيات (DI)

DI هو تطبيق عملي لـ Composition: تمرير dependencies من الخارج بدلاً من إنشائها داخلياً:

# بدون DI - الكلاس يخلق dependencies بنفسه (اقتران قوي) class OrderService: def __init__(self): self.logger = FileLogger() # مقرون بـ FileLogger! self.email = SendGridEmail() # مقرون بـ SendGrid! # مع DI - نمرر dependencies (اقتران ضعيف) class OrderService: def __init__(self, logger, email_service): self.logger = logger # أي Logger! self.email = email_service # أي EmailService! # يمكننا تبديل أي مكون بسهولة service = OrderService(ConsoleLogger(), SMTPEmail()) test_service = OrderService(MockLogger(), MockEmail()) # للاختبارات
🔍 Dependency Injection - حقن التبعيات

بدون DI، الكلاس يخلق dependencies بنفسه (FileLogger()، SendGridEmail()) مما يسبب اقتراناً قوياً ويصعب الاختبار. مع DI، نمرر dependencies عبر constructor. هذا يسمح بتبديل أي مكون بسهولة: استخدم ConsoleLogger في التطوير، FileLogger في الإنتاج، و MockLogger في الاختبارات. DI هو تطبيق عملي لمبدأ DIP (Dependency Inversion). أطر العمل مثل dependency_injector و FastAPI تستخدم DI بشكل مكثف.

قاعدة عامة: إذا احتجت 3+ مستويات وراثة، فتوقف وفكر في Composition بدلاً من ذلك. التركيب يجعل الكود أسهل للاختبار والصيانة.

14. Design Patterns - أنماط التصميم في بايثون

أنماط التصميم (Design Patterns) هي حلول قابلة لإعادة الاستخدام لمشاكل شائعة في البرمجة. إليك أهم الأنماط في سياق بايثون:

Factory Method & Abstract Factory - المصنع البسيط والمجرد

from abc import ABC, abstractmethod # Factory Method البسيط class PaymentGateway(ABC): @abstractmethod def charge(self, amount): pass class StripeGateway(PaymentGateway): def charge(self, amount): print(f'Stripe: charging ${amount}') class PayPalGateway(PaymentGateway): def charge(self, amount): print(f'PayPal: charging ${amount}') class PaymentFactory: _gateways = {'stripe': StripeGateway, 'paypal': PayPalGateway} @classmethod def create(cls, gateway_type): gateway_cls = cls._gateways.get(gateway_type) if not gateway_cls: raise ValueError(f'Unknown gateway: {gateway_type}') return gateway_cls() # الاستخدام gateway = PaymentFactory.create('stripe') gateway.charge(100)
🔍 Factory Method - نمط المصنع

نمط Factory Method يفصل عملية إنشاء الكائنات عن استخدامها. PaymentFactory يقرر أي نوع من PaymentGateway سيتم إنشاؤه بناءً على مفتاح (stripe أو paypal). المزايا: يمكن إضافة بوابات دفع جديدة بإضافة كلاس وربطه في الـ factory دون تغيير كود العميل. @classmethod يجعل factory method متاحاً بدون إنشاء كائن من الـ factory. هذا النمط شائع في أنظمة الدفع والتوصيل والتنسيقات المختلفة للبيانات.

# Abstract Factory - مصنع ينتج عائلات كاملة من المنتجات class Button(ABC): @abstractmethod def render(self): pass class Checkbox(ABC): @abstractmethod def render(self): pass class WindowsButton(Button): def render(self): return '[Windows Button]' class MacButton(Button): def render(self): return '[Mac Button]' class WindowsCheckbox(Checkbox): def render(self): return '[Windows Checkbox]' class MacCheckbox(Checkbox): def render(self): return '[Mac Checkbox]' class GUIFactory(ABC): # Abstract Factory @abstractmethod def create_button(self) -> Button: pass @abstractmethod def create_checkbox(self) -> Checkbox: pass class WindowsFactory(GUIFactory): def create_button(self): return WindowsButton() def create_checkbox(self): return WindowsCheckbox() class MacFactory(GUIFactory): def create_button(self): return MacButton() def create_checkbox(self): return MacCheckbox() # استخدام Abstract Factory - نبدل المنصة كلها بسهولة def create_ui(factory: GUIFactory): btn = factory.create_button() cb = factory.create_checkbox() return [btn.render(), cb.render()] print(create_ui(WindowsFactory())) # ['[Windows Button]', '[Windows Checkbox]'] print(create_ui(MacFactory())) # ['[Mac Button]', '[Mac Checkbox]']
🔍 Abstract Factory - مصنع ينتج عائلات كاملة

Abstract Factory يوسّع فكرة Factory Method لينتج عائلة كاملة من المنتجات المتوافقة. GUIFactory يحدد واجهة لإنشاء زراير و checkboxes. WindowsFactory ينتج زراً و checkbox بنمط Windows، و MacFactory بنمط Mac. هذا يضمن أن المنتجات من نفس العائلة متوافقة (لن تحصل على زر Windows مع checkbox Mac). تغيير المنصة يتم بتغيير factory واحد. هذا النمط يستخدم في مكتبات GUI عبر المنصات مثل Qt و wxPython.

Singleton Pattern - أنماط متعددة للتنفيذ

Singleton يضمن وجود نسخة واحدة فقط من الكلاس. في بايثون، هناك عدة طرق:

# الطريقة 1: module-level (الأبسط - والأفضل غالباً) # في ملف database.py: class Database: def __init__(self): self.connected = False def connect(self): ... db = Database() # هذه النسخة الوحيدة - تستورد من أي مكان # الاستخدام في ملف آخر: # from database import db # نفس النسخة دائماً!
🔍 Singleton - نمط الوحدة الواحدة (Module-level)

أبسط وأفضل طريقة للـ Singleton في بايثون: نسخة واحدة من الكلاس تُنشأ على مستوى module. عندما تستورد from database import db من أي مكان في التطبيق، تحصل على نفس النسخة. بايثون تضمن أن modules يتم تنفيذها مرة واحدة فقط (cached after first import). هذه الطريقة بسيطة، واضحة، ولا تحتاج لأي كود إضافي. مناسبة لمعظم حالات singleton مثل اتصالات قاعدة البيانات وإعدادات التطبيق (logging, configuration).

# الطريقة 2: __new__ Singleton (مفيد للكلاسات التي تحتاج وراثة) class Singleton: _instance = None def __new__(cls, *args, **kwargs): if cls._instance is None: cls._instance = super().__new__(cls) cls._instance._initialized = False return cls._instance def __init__(self): if not self._initialized: print('Initializing once') self._initialized = True s1 = Singleton() # Initializing once s2 = Singleton() # لا يطبع - نفس النسخة! print(s1 is s2) # True
🔍 Singleton عبر __new__ - التحكم في إنشاء الكائن

هذه الطريقة تستخدم __new__ للتحكم في إنشاء الكائن. __new__ يُستدعى قبل __init__ وهو المسؤول عن تخصيص الذاكرة. إذا كانت _instance موجودة، نعيدها دون إنشاء جديدة. العلم _initialized يمنع إعادة تهيئة الكائن في كل استدعاء. هذه الطريقة تحافظ على نوع الكائن (object identity) وتعمل بشكل جيد مع الوراثة. مقارنة بالطريقة module-level، تمنح تحكماً أكثر لكنها أكثر تعقيداً.

# الطريقة 3: Metaclass Singleton (يناسب الوراثة) class SingletonMeta(type): _instances = {} def __call__(cls, *args, **kwargs): if cls not in cls._instances: cls._instances[cls] = super().__call__(*args, **kwargs) return cls._instances[cls] class Config(metaclass=SingletonMeta): def __init__(self): print('Loading config...') c1 = Config() # Loading config... c2 = Config() # لا يطبع
🔍 Metaclass Singleton - التحكم في إنشاء النسخ

هذا metaclass يطبق نمط Singleton عبر override __call__ في الـ metaclass. الفرق عن class decorator أن metaclass يحافظ على نوع الكائن (type identity). __call__ في metaclass يُستدعى عند إنشاء كائن من الكلاس (Database()). يتحقق أولاً إذا كان هناك نسخة موجودة، وإلا ينشئ واحدة عبر super().__call__(). هذا يضمن نسخة واحدة لكل كلاس يستخدم هذا metaclass. هذه الطريقة هي الأفضل للـ singleton عندما تحتاج الوراثة أو الحفاظ على typing صحيح.

⚠️ متى تتجنب Singleton: يجعل الكود مقروناً بشدة (tight coupling) ويصعب اختباره. استخدم Dependency Injection مع Composition بدلاً من Singleton كلما أمكن. Singleton مفيد حقاً فقط للـ logging, configuration, connection pools.

Observer Pattern - نمط المراقب

from abc import ABC, abstractmethod class Observer(ABC): @abstractmethod def update(self, message): pass class EmailNotifier(Observer): def update(self, message): print(f'Email: {message}') class SMSNotifier(Observer): def update(self, message): print(f'SMS: {message}') class OrderService: def __init__(self): self._observers = [] def attach(self, observer): self._observers.append(observer) def _notify(self, message): for obs in self._observers: obs.update(message) def place_order(self, order): print(f'Processing order {order.id}') self._notify(f'Order {order.id} placed') # الاستخدام service = OrderService() service.attach(EmailNotifier()) service.attach(SMSNotifier()) service.place_order(order)
🔍 Observer Pattern - نمط المراقب

نمط Observer يعرّف علاقة واحد-إلى-متعدد بين الكائنات: عندما يتغير الكائن الرئيسي (Subject)، يتم إعلام جميع المراقبين (Observers) تلقائياً. OrderService هو Subject يسمح بإضافة Observers (EmailNotifier, SMSNotifier). عند تنفيذ طلب (place_order)، يستدعي _notify الذي يمرر الرسالة لكل Observer. هذا يفصل بين منطق العمل الأساسي وآليات الإعلام، ويسهل إضافة أنواع جديدة من الإشعارات دون تعديل OrderService.

Strategy Pattern - نمط الاستراتيجية

from abc import ABC, abstractmethod class DiscountStrategy(ABC): @abstractmethod def calculate(self, price): pass class NoDiscount(DiscountStrategy): def calculate(self, price): return price class PercentageDiscount(DiscountStrategy): def __init__(self, percent): self.percent = percent def calculate(self, price): return price * (1 - self.percent / 100) class FixedAmountDiscount(DiscountStrategy): def __init__(self, amount): self.amount = amount def calculate(self, price): return max(0, price - self.amount) class ShoppingCart: def __init__(self, strategy: DiscountStrategy): self.strategy = strategy # استراتيجية الخصم قابلة للتبديل self.items = [] def total(self): subtotal = sum(item.price for item in self.items) return self.strategy.calculate(subtotal) # نبدل الاستراتيجية بسهولة cart = ShoppingCart(PercentageDiscount(10)) # cart.strategy = FixedAmountDiscount(20) # تغيير الاستراتيجية في أي وقت!
🔍 Strategy Pattern - نمط الاستراتيجية

نمط Strategy يعرّف عائلة من الخوارزميات القابلة للتبديل. DiscountStrategy هو الـ interface المشترك، وكل استراتيجية (NoDiscount, PercentageDiscount, FixedAmountDiscount) تطبق طريقة حساب مختلفة. ShoppingCart يستقبل استراتيجية الخصم عبر constructor ويمكن تغييرها في أي وقت (cart.strategy = FixedAmountDiscount(20)). هذا يتبع مبدأ OCP: إضافة استراتيجية جديدة لا يتطلب تعديل ShoppingCart. Strategy Pattern يحقق "فصل الاهتمامات" (separation of concerns) بشكل جميل.

15. الوراثة المتعددة و MRO - تحليل عميق

تستخدم بايثون خوارزمية C3 Linearization (أو C3 superclass linearization) لتحديد ترتيب البحث (MRO). هذه الخوارزمية تضمن:

  • الكلاس الابن يأتي دائماً قبل الكلاس الأب
  • إذا ورث كلاس من عدة كلاسات، ترتيبهم في class definition يحدد الأولوية
  • الكلاس object دائماً في النهاية
# مثال معقد لـ MRO class A: def method(self): return 'A' class B(A): def method(self): return 'B' class C(A): def method(self): return 'C' class D(B, C): pass print(D.__mro__) # D -> B -> C -> A -> object # C3 Linearization: merge([B,C], [B,A,object], [C,A,object], [A,object]) # معقدة؟ فقط تذكر: child قبل parent، والترتيب الذي تحدده هو الأولوية
🔍 MRO المتقدم - C3 Linearization

هذا مثال معقد لـ MRO باستخدام الوراثة المتعددة. خوارزمية C3 Linearization المستخدمة من بايثون تضم أبعاداً ثلاثة: (1) الكلاس الابن يسبق الكلاس الأب. (2) ترتيب الكلاسات الأب في تعريف الكلاس يحدد الأولوية. (3) الـ object دائماً في النهاية. D(B, C) → MRO: D → B → C → A → object. لفهمها، تذكر أن الـ algorithm يبدأ بالكلاس الحالي، ثم يدمج (merge) قوائم MRO للكلاسات الأب بطريقة تحافظ على الترتيب النسبي. قاعدة عملية: child قبل parent، والـ first parent له أولوية.

# super() في الوراثة المتعددة (Cooperative Multiple Inheritance) class Base: def __init__(self, **kwargs): print('Base.__init__') super().__init__(**kwargs) class LogMixin: def __init__(self, **kwargs): print('LogMixin.__init__') super().__init__(**kwargs) class ValidateMixin: def __init__(self, **kwargs): print('ValidateMixin.__init__') super().__init__(**kwargs) class User(LogMixin, ValidateMixin, Base): def __init__(self, name, **kwargs): print('User.__init__') super().__init__(**kwargs) self.name = name # MRO: User -> LogMixin -> ValidateMixin -> Base -> object # كل super() يستدعي التالي في السلسلة! user = User(name='Ahmed') # User.__init__ # LogMixin.__init__ # ValidateMixin.__init__ # Base.__init__
🔍 Cooperative Multiple Inheritance - super() مع **kwargs

في الوراثة المتعددة التعاونية، كل كلاس يستخدم super().__init__(**kwargs) بدلاً من استدعاء كلاس أب محدد. هذا يسمح لكل كلاس بأخذ ما يحتاجه من **kwargs وتمرير الباقي للكلاس التالي في سلسلة MRO. الترتيب: User → LogMixin → ValidateMixin → Base → object. كل كلاس يطبع رسالة ويستدعي super(). النتيجة: يتم استدعاء جميع constructors بالترتيب الصحيح. نصيحة ذهبية: استخدم دائماً **kwargs مع super() في الوراثة المتعددة لتجنب المشاكل.

نصيحة ذهبية: في الوراثة المتعددة، استخدم **kwargs و super() دائماً مع **kwargs. هذا يسمح لكل كلاس بأخذ ما يحتاجه وتمرير الباقي للكلاس التالي في MRO.

16. أنماط الأداء والذاكرة

فهم كيفية إدارة بايثون للذاكرة يساعدك على كتابة كود OOP أسرع وأخف:

__slots__ - تحسين الذاكرة

كل كائن بايثون يخزن خصائصه في __dict__ (dict) الذي يستهلك ذاكرة إضافية. __slots__ يمنع إنشاء __dict__ ويخزن الخصائص في tuple، مما يوفر ذاكرة بشكل كبير للكلاسات ذات آلاف النسخ:

import sys class PointNoSlots: def __init__(self, x, y): self.x = x; self.y = y class PointWithSlots: __slots__ = ('x', 'y') def __init__(self, x, y): self.x = x; self.y = y p1 = PointNoSlots(1, 2) p2 = PointWithSlots(1, 2) print(sys.getsizeof(p1.__dict__)) # يحتوي __dict__ print(sys.getsizeof(p2)) # بدون __dict__ - أصغر! # __slots__ يمنع إضافة خصائص جديدة # p2.z = 3 # AttributeError: 'PointWithSlots' has no attribute 'z'
🔍 __slots__ في Point - توفير الذاكرة

__slots__ يمنع إنشاء __dict__ لكل كائن ويخزن الخصائص في array ذي حجم ثابت، مما يقلل استهلاك الذاكرة بشكل كبير (يصل إلى 40-50% توفيراً). كل كائن بايثون يحمل __dict__ وهو قاموس يستهلك ذاكرة إضافية (hashing overhead). باستخدام __slots__، نحدد الخصائص المسموح بها مسبقاً ولا يمكن إضافة خصائص جديدة ديناميكياً (محاولة تعيين p.z = 5 سترفع AttributeError). هذا مفيد جداً في الكلاسات التي تنشئ آلاف النسخ مثل تمثيل البيانات في machine learning أو معالجة datasets ضخمة.

متى تستخدم __slots__: عندما يكون لديك آلاف أو ملايين النسخ من الكلاس (مثل معالجة بيانات ضخمة). للكلاسات القليلة، لا داعي.

__weakref__ - المراجع الضعيفة

المراجع الضعيفة تسمح بالإشارة إلى كائن دون زيادة عداد المراجع (reference count). مفيدة للـ caches و avoid circular references:

import weakref class Cache: _cache = weakref.WeakValueDictionary() @classmethod def get(cls, key): return cls._cache.get(key) @classmethod def set(cls, key, value): cls._cache[key] = value # المراجع الضعيفة تسمح بجمع القمامة (garbage collection) # حتى لو كان الكائن في الـ cache!
🔍 __weakref__ - المراجع الضعيفة لإدارة الذاكرة

المراجع الضعيفة (weak references) تسمح بالإشارة إلى كائن دون زيادة عداد المراجع الخاص به. هذا يسمح لـ garbage collector بجمع الكائن حتى عندما يكون هناك مرجع ضعيف指向ه. WeakValueDictionary هو قاموس يخزن قيماً بمراجع ضعيفة: إذا لم يعد هناك مراجع قوية للقيمة، يتم حذفها تلقائياً من القاموس. مفيد جداً للـ caches: يمكن التخزين المؤقت للبيانات دون منعها من التحرير عندما لا تكون قيد الاستخدام. أيضاً مفيد لتجنب circular references التي تسبب تسرب الذاكرة.

Cached Properties - خصائص مخزنة مؤقتاً

from functools import cached_property class DataAnalyzer: def __init__(self, data): self.data = data @cached_property def expensive_computation(self): # تُحسب مرة واحدة فقط، ثم تخزن في الذاكرة print('Computing...') return sum(x ** 2 for x in self.data) da = DataAnalyzer([1, 2, 3, 4, 5]) print(da.expensive_computation) # Computing... 55 print(da.expensive_computation) # 55 (بدون إعادة حساب!)
🔍 @cached_property - التخزين المؤقت للخصائص المحسوبة

@cached_property من functools يجمع بين @property و caching. في أول وصول للخاصية، تُحسب القيمة وتُخزّن. في المرات التالية، تُعاد القيمة المخزنة مباشرة بدون إعادة حساب. هذا مثالي للـ expensive operations (حسابات معقدة، استعلامات قاعدة بيانات، عمليات I/O) التي لا تتغير نتائجها. المثال يوضح أن expensive_computation تُطبع "Computing..." مرة واحدة فقط في أول استدعاء. الفرق عن الـ @property العادي أن caching يجنب إعادة الحساب في كل مرة.

17. الاستثناءات المخصصة (Custom Exceptions)

إنشاء تسلسل هرمي للاستثناءات الخاصة بك يجعل معالجة الأخطاء أكثر دقة:

class AppError(Exception): """Base exception for our application""" def __init__(self, message, code=None): super().__init__(message) self.code = code class ValidationError(AppError): """خطأ في التحقق من صحة البيانات""" def __init__(self, field, message): super().__init__(f'Validation failed for {field}: {message}', code=400) self.field = field class AuthenticationError(AppError): def __init__(self, message='Authentication failed'): super().__init__(message, code=401) class NotFoundError(AppError): def __init__(self, resource, id): super().__init__(f'{resource} with id {id} not found', code=404) # الاستخدام - معالجة دقيقة حسب النوع try: user = find_user(42) except NotFoundError as e: print(f'Not found: {e}') # ليس خطأ حقيقياً except ValidationError as e: print(f'Field {e.field}: {e}') # خطأ في الإدخال except AppError as e: print(f'App error {e.code}: {e}') # أي خطأ آخر
🔍 Custom Exceptions - الاستثناءات المخصصة

إنشاء تسلسل هرمي للاستثناءات (AppError → ValidationError, AuthenticationError, NotFoundError) يسمح بمعالجة دقيقة للأخطاء. AppError كـ base يرث من Exception ويضيف خاصية code. كل استثناء متخصص يضيف خصائص إضافية (مثل field في ValidationError). عند المعالجة، يمكن استخدام except متعدد للتعامل مع كل نوع بشكل مختلف: NotFoundError مجرد إعلام، ValidationError خطأ في الإدخال، إلخ. هذا أفضل من رفع Exception عام والتحقق من النص داخله.

نصيحة: ابدأ دائماً باستثناء Base واحد (مثل AppError)، واجعله يرث من Exception (وليس BaseException). هذا يسمح لك بالقبض على كل أخطاء تطبيقك بقاعدة واحدة.

18. أنواع متقدمة من الخصائص

Computed @property مع caching يدوي

class Order: def __init__(self, items, tax_rate): self.items = items self.tax_rate = tax_rate self._total = None # cache @property def total(self): if self._total is None: subtotal = sum(item.price * item.qty for item in self.items) self._total = subtotal * (1 + self.tax_rate) return self._total @total.setter def total(self, value): raise AttributeError('Total is read-only')
🔍 Computed @property مع Caching يدوي

هذا example يوضح كيفية تنفيذ التخزين المؤقت يدوياً للـ @property. خاصية total تحسب المجموع مع الضريبة. المتغير _total يُخزّن النتيجة مؤقتاً: إذا كان None تعاد الحسابات، وإلا تعاد القيمة المخزنة. الـ setter يرفع AttributeError لأن total للقراءة فقط (read-only). هذا النمط مفيد عندما تحتاج تحكماً أكثر من @cached_property (مثلاً، تحتاج إبطال cache عند تغيير البيانات). في الإصدارات الحديثة، @cached_property أبسط لكن manual caching يعطيك مرونة أكبر.

__init_subclass__ - Hook عند إنشاء كلاس ابن

class PluginBase: _plugins = {} def __init_subclass__(cls, **kwargs): super().__init_subclass__(**kwargs) # يُستدعى تلقائياً عند تعريف أي كلاس يرث من PluginBase name = cls.__name__ cls._plugins[name] = cls print(f'Plugin registered: {name}') @classmethod def run_all(cls): for name, plugin in cls._plugins.items(): print(f'Running {name}') plugin().execute() # هذه الكلاسات تُسجل تلقائياً! class EmailPlugin(PluginBase): def execute(self): print('Sending email...') class SMSPlugin(PluginBase): def execute(self): print('Sending SMS...') PluginBase.run_all() # Running EmailPlugin # Running SMSPlugin
🔍 __init_subclass__ - تسجيل تلقائي للكلاسات الابنة

__init_subclass__ هو hook يُستدعى تلقائياً كلما تم تعريف كلاس جديد يرث من الكلاس الحالي. هنا، كل كلاس يرث من PluginBase يتم تسجيله تلقائياً في _plugins dictionary باسمه. هذا يلغي الحاجة لتسجيل الـ plugins يدوياً. الميثود run_all() يمر على كل الـ plugins المسجلة وينفذها. هذه تقنية قوية تستخدم في أطر العمل لتجميع المكونات تلقائياً (auto-discovery). __init_subclass__ أسهل وأكثر أماناً من metaclasses في كثير من الحالات.

19. خلاصة ونصائح للاحتراف

📝 خلاصة OOP في بايثون - دليل سريع

  • كلاس: مخطط لإنشاء الكائنات، يحتوي attributes و methods
  • Instance Methods: تأخذ self - تتعامل مع بيانات الكائن
  • Class Methods (@classmethod): تأخذ cls - للـ alternative constructors والـ class state
  • Static Methods (@staticmethod): لا تأخذ self/cls - دوال عادية داخل namespace الكلاس
  • @property: للتحكم في الوصول للخصائص مع caching (@cached_property)
  • وراثة: super(), MRO (C3 Linearization), multiple inheritance
  • تعدد الأشكال: Duck typing, ABC, protocols (@runtime_checkable)
  • Dunder Methods: __init__, __str__, __repr__, __eq__, __hash__, __call__, __enter__/__exit__, __getitem__, __iter__
  • Decorators: @wraps, decorated with/without args, class decorators, stacking
  • Descriptors: __get__/__set__/__delete__ - data vs non-data
  • Metaclasses: type(), __new__, __call__ في metaclass، __init_subclass__
  • __slots__: توفير ذاكرة للكلاسات ذات آلاف النسخ
  • SOLID: 5 مبادئ لكود قابل للصيانة والتوسع
  • Design Patterns: Factory, Observer, Strategy
  • Context Managers: __enter__/__exit__ أو @contextmanager
  • Custom Exceptions: تسلسل هرمي خاص بتطبيقك
⚠️ أخطاء شائعة يجب تجنبها:
  • استخدام mutable objects كـ default arguments (def __init__(self, items=[])) - استخدم None بدلاً من ذلك
  • نسيان استدعاء super().__init__() في الكلاسات المشتقة
  • استخدام الوراثة المتعددة بدون فهم MRO - قد يؤدي إلى نتائج غير متوقعة
  • إنشاء getter/setter بدون سبب حقيقي - استخدم @property فقط عندما تحتاج تحكم/logic
  • تعديل الكلاس الأب بطريقة تؤثر على الكلاسات الابن (tight coupling)
  • استخدام type() للتحقق من الأنواع - استخدم isinstance() بدلاً من ذلك
