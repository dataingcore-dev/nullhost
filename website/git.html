Git/GitHub - التحكم بالنسخ

Version Control — إدارة تاريخ مشاريعك البرمجية بذكاء

Git هو أقوى وأشهر نظام للتحكم بالنسخ (Version Control System) في العالم. يُستخدم لتتبع التغييرات في ملفات مشروعك، والعودة للإصدارات السابقة، والعمل مع فريق دون أن تتعارض تعديلاتكم. GitHub هي منصة سحابية تستضيف مستودعات Git وتضيف أدوات تعاون إضافية مثل Pull Requests و Issues. هذا الدليل مخصص لمطوري بايثون لتعلم Git و GitHub من الصفر.

1. مقدمة عن Git و GitHub

التحكم بالنسخ (Version Control) هو نظام يُسجّل التغييرات التي تطرأ على ملفاتك عبر الزمن. يسمح لك بالعودة لإصدارات سابقة، معرفة من أجرى تغييراً معيناً، وتجربة أفكار جديدة في فروع منفصلة دون التأثير على الكود المستقر.

Git — نظام تحكم بالنسخ موزّع (Distributed VCS). كل مطور لديه نسخة كاملة من تاريخ المشروع على جهازه، مما يسمح بالعمل دون اتصال بالإنترنت. Git سريع جداً، ويستخدمه ملايين المطورين حول العالم.

GitHub — خدمة سحابية تستضيف مستودعات Git. تضيف GitHub أدوات اجتماعية وتعاونية مثل: Pull Requests (طلبات الدمج)، Issues (تتبع المشاكل)، Code Review (مراجعة الكود)، GitHub Actions (التكامل المستمر)، و GitHub Pages (نشر المواقع). الفرق بين Git و GitHub: Git هو الأداة، GitHub هو المنصة.

💡 لماذا Git لمطوري بايثون؟ كل مشروع بايثون — من سكربت بسيط إلى إطار ويب كبير — يحتاج إلى Git. أنت تحتاجه لتجربة أفكار جديدة في branches، للعودة عن أخطاء، وللتعاون مع فريقك عبر GitHub أو GitLab أو Bitbucket. أغلب وظائف بايثون تُنشر عبر GitHub.

2. تثبيت Git وتكوينه

لتثبيت Git على نظام Linux (مثل Ubuntu/Debian):

sudo apt update sudo apt install git -y git --version
🔍 شرح الكود:

الأمر الأول يُحدّث قائمة الحزم المتاحة. الثاني يُثبّت Git مع العلم -y لتأكيد التثبيت تلقائياً. الثالث يتحقق من إصدار Git المثبّت — إذا رأيت رقماً مثل git version 2.34.1، فالتثبيت نجح.

على Windows، حمّل المثبّت من git-scm.com. على macOS، استخدم brew install git.

بعد التثبيت، يجب تكوين اسمك وبريدك الإلكتروني — هذه المعلومات تُرفق مع كل commit:

git config --global user.name "Your Name" git config --global user.email "you@example.com" git config --list
🔍 شرح الكود:

الأمر الأول يضبط اسمك العالمي — كل commits ستُنسَب لهذا الاسم. الثاني يضبط بريدك الإلكتروني (يُفضّل أن يكون نفسه المسجّل في GitHub). الثالث يعرض جميع الإعدادات الحالية لتأكيد التهيئة الصحيحة. --global يعني أن هذه الإعدادات تطبق على كل مستودعات Git في جهازك.

3. إنشاء مستودع (Repository)

المستودع (Repository أو "repo") هو مجلد يحتوي على مشروعك وكل تاريخ Git له. لإنشاء مستودع جديد:

mkdir my-python-project cd my-python-project git init
🔍 شرح الكود:

mkdir تنشئ مجلداً جديداً باسم my-python-project. cd تنتقل إليه. git init تنشئ مستودع Git جديد — تنشئ مجلداً مخفياً باسم .git يحتوي على كل بيانات Git. الآن هذا المجلد هو مستودع Git رسمي.

بعد git init، ستجد مجلداً مخفياً باسم .git (أظهره بالأمر ls -la). هذا المجلد هوعمود Git الفقري — يحتوي على كل commits والفروع والتاريخ. لا تُحرّره يدوياً أبداً!

لمعرفة حالة المستودع الحالية — أي الملفات التي تم تعديلها، والتي في منطقة التجهيز (staging area)، وأيها غير متتبّع (untracked):

git status
🔍 شرح الكود:

git status هو أكثر أوامر Git استخداماً. يظهر لك الملفات التي عدّلتها ولم تُضفها بعد، الملفات في منطقة التجهيز (staged)، والملفات الجديدة التي لم يتتبّعها Git بعد (untracked). استخدمه دائماً قبل commit لتعرف بالضبط ما الذي ستُسجّله.

4. دورة حياة الملفات (Status Lifecycle)

كل ملف في مستودع Git يمر بحالات محددة. فهم دورة الحياة هذه هو المفتاح لفهم Git:

  • Untracked (غير متتبّع): Git لا يعلم بوجود هذا الملف بعد. أنشأته للتو ولم تُضفه.
  • Staged (مُجهّز): أضفت الملف إلى منطقة التجهيز بانتظار commit.
  • Committed (مُسجّل): تم حفظ الملف في تاريخ Git.
  • Modified (مُعدّل): الملف مُسجّل مسبقاً لكن أجريت تغييرات جديدة لم تُجهّز بعد.

دورة الحياة الأساسية تعمل هكذا: Working Directory (مجلد العمل) ← Staging Area (منطقة التجهيز) ← Repository (المستودع). المجلد حيث تعمل هو Working Directory. عندما تُضيف ملفاً بـ git add، ينتقل إلى Staging Area. عندما تنفذ git commit، يُسجّل في Repository.

# إنشاء ملف بايثون جديد echo 'print("Hello, Git!")' > main.py # الملف الآن Untracked — Git لا يتتبعه بعد git status # إضافة الملف إلى منطقة التجهيز (Staging Area) git add main.py # الملف الآن Staged — جاهز للcommit git status # تسجيل الملف في المستودع git commit -m "أضف ملف main.py الأولي"
🔍 شرح الكود:

أنشأنا ملف main.py يحتوي على أمر print. git status يُظهره كـ Untracked. git add main.py يُحرّكه إلى Staging Area — الآن Git يتتبّعه. git commit -m "..." يُسجّله رسمياً في تاريخ المستودع مع رسالة تصف التغيير. بعد commit، يصبح الملف نظيفاً (clean) حتى تُعدّله مرة أخرى.

5. git add و git commit

git add ينقل الملفات من مجلد العمل إلى منطقة التجهيز. يمكنك إضافة ملف واحد أو عدة ملفات أو كل الملفات:

# إضافة ملف واحد git add main.py # إضافة مجلد كامل git add utils/ # إضافة كل الملفات (الأكثر استخداماً) git add . # إضافة مع حذف ملفات من staging git add -A
🔍 شرح الكود:

git add main.py يضيف ملفاً واحداً فقط. git add utils/ يضيف كل الملفات داخل مجلد utils. git add . يضيف كل الملفات الجديدة والمعدّلة في المجلد الحالي (لكن لا يحذف الملفات المحذوفة من staging). git add -A يفعل كل شيء — إضافات وتعديلات وحذف.

git commit يُسجّل لقطة (snapshot) لكل الملفات في منطقة التجهيز. كل commit له معرّف فريد (SHA-1 hash) ورسالة تصف التغيير:

# commit بسيط git commit -m "إصلاح خطأ في حساب الضرائب" # commit مع وصف أطول (يفتح محرر نصوص) git commit
🔍 شرح الكود:

git commit -m "..." هو اختصار — تكتب الرسالة مباشرة في الأمر. إذا كتبت git commit بدون -m، سيفتح Git محرر النصوص الافتراضي (مثل Nano أو Vim) لتكتب رسالة أطول. اكتب commit messages واضحة تبدأ بفعل أمر (Fix, Add, Update, Remove).

لمشاهدة تاريخ commits:

# عرض كل commits git log # عرض سطر واحد لكل commit git log --oneline # عرض شجرة الفروع git log --oneline --graph --all
🔍 شرح الكود:

git log يعرض كل commits بالتفصيل — المؤلف، التاريخ، الرسالة، و SHA-1 hash. --oneline يختصر كل commit في سطر واحد (الـ hash المختصر + الرسالة). --graph --all يعرض شجرة الفروع بشكل رسومي باستخدام خطوط ASCII — مفيد جداً لفهم تاريخ المشروع.

💡 نصائح لكتابة Commit Messages جيدة: اكتب الرسالة باللغة الإنجليزية (الأفضل للمشاريع العالمية) أو العربية. ابدأ بفعل: "Add", "Fix", "Update", "Remove", "Refactor". أول سطر أقل من 50 حرفاً. اترك سطراً فارغاً ثم اكتب شرحاً أطول إن لزم الأمر. مثال: "Add user authentication with JWT tokens".

6. الفروع (Branches)

الفروع (Branches) هي أقوى ميزة في Git. الفرع هو مؤشّر متحرّك إلى commit معيّن. عندما تنشئ فرعاً جديداً، تنسخ تاريخ المشروع عند تلك النقطة وتعمل عليه بشكل مستقل — دون التأثير على الفرع الرئيسي main.

# عرض الفروع الموجودة git branch # إنشاء فرع جديد git branch feature-login # التبديل إلى فرع آخر git checkout feature-login # إنشاء وتبديل في أمر واحد (طريقة حديثة) git checkout -b feature-api # الطريقة الحديثة (Git 2.23+) git switch -b feature-api
🔍 شرح الكود:

git branch يعرض الفروع — الفرع النشط (الحالي) عليه علامة *. git branch feature-login ينشئ فرعاً جديداً لكنك تبقى في الفرع الحالي. git checkout feature-login يُبدّلك إليه. git checkout -b feature-api ينشئ ويُبدّل فوراً. git switch -b هي الطريقة الأحدث (Git 2.23+) وتعمل بنفس المنطق.

تخيّل main كالعمود الفقري للمشروع. عندما تريد إضافة ميزة جديدة (login, API, إلخ)، أنشئ فرعاً من main، اعمل بهدوء، ثم ادمجه مرة أخرى. هذا يحمي main من الكود المكسور.

# إنشاء فرع وتعديل ملف git switch -c feature-pagination echo 'def paginate(): pass' >> utils.py git add . && git commit -m "Add pagination utility" # العودة إلى main git switch main # حذف فرع (بعد الدمج) git branch -d feature-pagination
🔍 شرح الكود:

أنشأنا فرعاً جديداً feature-pagination وانتقلنا إليه. أضفنا دالة بايثون إلى ملف utils.py، ثم أضفنا الملف إلى staging (git add .) وسجّلنا commit. git switch main يعيدنا للفرع الرئيسي — الملف سيعود لحالته قبل التعديل (الفرعان مستقلان). بعد الدمج، نحذف الفرع بـ git branch -d.

7. الدمج (Merge)

الدمج (Merge) يجمع التغييرات من فرع إلى آخر. مثلاً، بعد الانتهاء من تطوير ميزة في فرع منفصل، تدمجها إلى main:

# تأكد أنك في الفرع الهدف (main) git switch main # ادمج فرع الميزة إلى main git merge feature-pagination
🔍 شرح الكود:

أولاً، تأكد أنك في الفرع الذي تريد الدمج إليه — هنا main. ثم git merge feature-pagination يدمج commits من feature-pagination إلى main. إذا لم يكن هناك تعارضات، سترى رسالة "Fast-forward" أو "Merge made by the 'ort' strategy".

هناك نوعان رئيسيان من الدمج:

  • Fast-Forward Merge: يحدث عندما يكون الفرع الهدف (main) لم يتغير منذ إنشاء الفرع الجديد. Git ببساطة يُحرّك المؤشر إلى الأمام.
  • 3-Way Merge: يحدث عندما يكون الفرعان (main و feature) قد تطورا بشكل منفصل. ينشئ Git commit دمج جديد برأسَين (merge commit).

تعارضات الدمج (Merge Conflicts): عندما يعدّل الفرعان نفس السطر في نفس الملف بشكل مختلف، لا يستطيع Git الدمج تلقائياً. يجب حل التعارض يدوياً:

# Git سيخبرك بوجود تعارض git merge feature-login # CONFLICT (content): Merge conflict in main.py # افتح الملف وابحث عن علامات Git: # <<<<<<< HEAD # print("Hello from main") # ======= # print("Hello from feature") # >>>>>>> feature-login # عدّل الملف — احتفظ بما تريد واحذف العلامات # ثم أضف و commit كالمعتاد: git add . git commit -m "Resolve merge conflict in main.py"
🔍 شرح الكود:

عند حدوث تعارض، Git يُوقف الدمج ويُدرج علامات في الملف: <<<<<<< HEAD (نسخة الفرع الحالي)، ======= (فاصل)، >>>>>>> feature-login (نسخة الفرع المدمَج). احتفظ بالكود الصحيح، احذف العلامات، ثم git add و git commit لإكمال الدمج.

8. GitHub - العمل عن بعد

GitHub يضيف بُعداً جديداً: المستودع البعيد (Remote Repository). المستودع المحلي على جهازك والمستودع البعيد على GitHub — يمكنك مزامنتهما:

# إنشاء مستودع على GitHub أولاً، ثم ربطه محلياً git remote add origin https://github.com/username/my-project.git # دفع commits المحلية إلى GitHub git push -u origin main # في المرات التالية (بعد -u أول مرة): git push
🔍 شرح الكود:

git remote add origin ... يُضيف المستودع البعيد باسم origin (التقليد). git push -u origin main يدفع commits المحلية إلى GitHub وينشئ رابط تتبّع بين الفرع المحلي main والفرع البعيد origin/main. العلم -u (upstream) يسمح لك باستخدام git push فقط في المستقبل.

لنسخ مستودع من GitHub إلى جهازك (مثل مشروع مفتوح المصدر):

# clone — نسخة كاملة من المستودع git clone https://github.com/username/project.git # git pull = git fetch + git merge (جلب التحديثات) git pull
🔍 شرح الكود:

git clone يُنشئ نسخة محلية كاملة من أي مستودع على GitHub — مع كل الفروع والتاريخ. git pull يجلب (fetch) التغييرات الجديدة من GitHub ويدمجها (merge) مع فرعك المحلي في خطوة واحدة. استخدم git pull يومياً قبل بدء العمل للحصول على آخر التغييرات.

git pull هو اختصار لخطوتين: git fetch (يجلب التغييرات البعيدة بدون دمج) و git merge (يدمجها مع فرعك المحلي). git fetch آمن — يمكنك مراجعة التغييرات ثم git merge يدوياً لاحقاً.

9. Pull Requests

Pull Request (PR) أو "طلب دمج" هو طريقة GitHub لاقتراح تغييرات على مشروع. بدلاً من دفع الكود مباشرة إلى main، تنشئ فرعاً، تدفعه إلى GitHub، وتفتح PR لمراجعته:

# سير عمل PR الكامل: # 1. حدّث فرعك المحلي git switch main git pull # 2. أنشئ فرعاً للميزة الجديدة git switch -c feature-dark-mode # 3. اعمل على الكود echo 'DARK_MODE = True' > config.py git add . && git commit -m "Add dark mode config" # 4. ادفع الفرع إلى GitHub git push -u origin feature-dark-mode # 5. اذهب إلى GitHub وافتح Pull Request # 6. يفحص الزملاء الكود ويتركون comments # 7. بعد الموافقة — اضغط "Merge PR" على GitHub # 8. احذف الفرع بعد الدمج: git switch main git pull git branch -d feature-dark-mode
🔍 شرح الكود:

هذا هو سير العمل الأساسي لأي مشروع تعاوني على GitHub. أنشئ فرعاً جديداً لكل مهمة، اكتب الكود واعمل commits، ادفع الفرع إلى GitHub، افتح PR، انتظر المراجعة، ثم ادمج. العلم -u يضبط upstream tracking. بعد الدمج، احذف الفرع محلياً لتجنب تراكم الفروع القديمة.

Fork هو نسخة من مستودع شخص آخر تحت حسابك على GitHub. تستخدم Fork للمساهمة في مشاريع مفتوحة المصدر: fork ← clone ← أنشئ فرعاً ← عدّل ← ادفع ← افتح PR ← يدمج صاحب المشروع تغييرك.

💡 نصائح PR: اجعل PR واحداً صغيراً ومركّزاً (مهمة واحدة فقط). اكتب وصفاً واضحاً يشرح "ماذا" و "لماذا". أضف Screenshots إن أمكن. استجب لتعليقات المراجعين بأدب. استخدم git push --force بحذر شديد بعد PR.

10. Rebase و Reset

git rebase هو بديل للدمج — يُعيد تطبيق commits من فرعك على قمة فرع آخر، مما يُنتج تاريخاً خطياً نظيفاً. الفرق بين rebase و merge:

# قبل rebase: feature مبني على commit قديم # main: A---B---C # feature: D---E # rebase: يضع commits D و E فوق C git switch feature git rebase main # النتيجة: main: A---B---C # feature: D'---E' # ثم fast-forward merge إلى main git switch main git merge feature
🔍 شرح الكود:

git rebase main يأخذ commits الفريدة في فرع feature (D و E) ويُعيد تطبيقها فوق أحدث commit في main (C). النتيجة: تاريخ خطي نقي — وكأنك طورت الميزة كلها بعد C. يحذّر: لا تستخدم rebase على فروع يشارك فيها آخرون (public branches) لأنه يُعيد كتابة التاريخ.

git reset يُحرّك مؤشر HEAD إلى commit سابق ويمكنه تعديل منطقة التجهيز ومجلد العمل بثلاث طرق:

# --soft: يحرّك HEAD فقط — staging و workdir كما هما git reset --soft HEAD~1 # --mixed (الافتراضي): يحرّك HEAD ويُفرّغ staging — workdir كما هو git reset --mixed HEAD~1 # --hard: يحرّك HEAD ويُفرّغ staging ويُعيد workdir — احذر! لا رجعة فيه git reset --hard HEAD~1
🔍 شرح الكود:

--soft تُرجع commit لكن تترك كل ملفاتك في Staging Area — كأنك أزلت commit فقط. --mixed (الافتراضي) تُرجع commit وتُخرج الملفات من staging (تبقى معدّلة في workdir). --hard تُرجع commit وتُعيد workdir لحالة ذلك commit — أي تغييرات غير مسجلة تُفقد للأبد! استخدم --hard بحذر شديد.

git revert هو البديل الآمن للـ reset. بدلاً من محو التاريخ، ينشئ commit جديداً يعاكس commit معيّن — آمن جداً للفروع العامة:

# إلغاء commit معيّن بطريقة آمنة git revert HEAD # إلغاء commit معيّن بالـ hash git revert a1b2c3d # الفرق: reset يحذف التاريخ، revert يضيف commit جديد يعاكسه
🔍 شرح الكود:

git revert HEAD ينشئ commit جديداً يلغي تغييرات آخر commit — history لا يتغير، فقط يُضاف commit معاكس. هذا آمن للفروع المشتركة (public branches) لأنك لا تُعيد كتابة التاريخ. استخدم revert دائماً في الفرق، و reset فقط للفروع المحلية التي لم تشاركها.

11. .gitignore و Best Practices

ملف .gitignore يخبر Git بتجاهل ملفات ومجلدات معيّنة — لا تُضاف أبداً إلى المستودع. هذا ضروري لمشاريع بايثون:

# .gitignore — Python project # Byte-compiled __pycache__/ *.py[cod] *.pyo # Virtual environment venv/ env/ .venv/ # IDE files .vscode/ .idea/ # Environment variables .env .env.local # Distribution / packaging dist/ build/ *.egg-info/ # OS files .DS_Store Thumbs.db # Jupyter Notebook checkpoints .ipynb_checkpoints/
🔍 شرح الكود:

كل سطر هو نمط تجاهل. __pycache__/ يتجاهل المجلد الذي يحتوي على ملفات البايت كود المترجمة. *.pyc يتجاهل كل ملفات البايت كود. venv/ يتجاهل البيئة الافتراضية — لا داعي لمشاركتها. .env يتجاهل المتغيرات السرية. .vscode/ يتجاهل إعدادات المحرّر. أضف ملف .gitignore في أول commit لمشروعك.

يمكنك أيضاً إنشاء .gitignore عالمي يطبق على كل مستودعاتك:

git config --global core.excludesfile ~/.gitignore_global
🔍 شرح الكود:

هذا الأمر يُخبر Git أن ملف ~/.gitignore_global هو ملف تجاهل عالمي — أنشئه بنفس صيغة .gitignore. كل أنماطه ستطبق تلقائياً على كل مستودعات Git في جهازك. مثالي لأنماط نظام التشغيل (.DS_Store) أو المحرّر (.vscode/).

إرشادات مهمة (Best Practices):

  • تسمية الفروع: استخدم أسماء وصفية بالإنگليزية: feature/login, bugfix/api-crash, hotfix/security-patch
  • Commit Messages: ابدأ بفعل (Add, Fix, Update, Remove, Refactor). لا تتجاوز 50 حرفاً في السطر الأول. اشرح "لماذا" في السطور التالية.
  • Commits صغيرة: commit واحد لكل تغيير منطقي. لا تنتظر حتى تتراكم 50 ملفاً.
  • لا تcommit ملفات: تحتوي على secrets, tokens, كلمات المرور, أو __pycache__/
  • اختبر قبل الدمج: شغّل اختبارات بايثون (pytest) قبل فتح PR.
  • استخدم Conventional Commits: feat: add user login, fix: correct tax calculation, docs: update README
💡 نمط Conventional Commits: feat: ... (ميزة جديدة) fix: ... (إصلاح خطأ) docs: ... (توثيق) refactor: ... (إعادة هيكلة بدون تغيير وظيفي) test: ... (إضافة اختبارات) chore: ... (مهام روتينية). هذا النمط يُستخدم مع Semantic Versioning وأدوات الـ Changelog.

12. التعامل مع الأخطاء (Troubleshooting)

حتى أفضل المطورين يقعون في أخطاء مع Git. إليك أدوات وحلول للمشاكل الشائعة:

git stash — تخزين التغييرات مؤقتاً

git stash يُخزّن تغييراتك غير الملتزمة (unstaged) مؤقتاً ويُعيد مجلد العمل لحالة نظيفة. مفيد جداً عندما تحتاج للتبديل إلى فرع آخر بسرعة:

# حفظ التغييرات غير الملتزمة مؤقتاً git stash # تطبيق آخر stash وحذفه git stash pop # تطبيق آخر stash بدون حذفه git stash apply # عرض قائمة stashes git stash list # stash مع رسالة git stash push -m "Work in progress on login feature" # stash untracked files أيضاً git stash -u
🔍 شرح الكود:

git stash يحفظ تغييراتك (الملفات المعدّلة وغير الملتزمة) في كومة (stack) ويعيد workdir لحالة آخر commit. git stash pop يُطبّق آخر stash ويزيله من الكومة. git stash apply يُطبّقه دون حذف. stash list يعرض الـ stashes المحفوظة. استخدم stash للتبديل السريع بين المهام دون فقدان التغييرات.

git diff — عرض التغييرات

git diff يُظهر الفروقات بين حالات الملفات:

# الفرق بين workdir و staging (تغييرات غير مضافة) git diff # الفرق بين staging وآخر commit (ما سيُسجّل) git diff --staged # الفرق بين commit معين و workdir git diff HEAD~2 # الفرق بين فرعين git diff main..feature
🔍 شرح الكود:

git diff يُظهر الأسطر المضافة (بالأخضر) والمحذوفة (بالأحمر) بالمقارنة مع staging. --staged يُظهر ما في staging مقابل آخر commit — أي ما سيُسجّله الـ commit القادم. diff main..feature يُظهر كل الفروقات بين فرعين. هذا الأمر هو صديقك لفهم "ماذا تغيّر بالضبط؟"

git log --all --graph — رؤية شاملة

هذا الأمر يعرض شجرة commits كاملة بكل الفروع:

git log --oneline --graph --all --decorate
🔍 شرح الكود:

هذا هو الأمر الأقوى لاستكشاف تاريخ المشروع. --oneline يختصر كل commit بسطر. --graph يرسم شجرة الفروع بخطوط ASCII. --all يظهر كل الفروع (حتى المحذوفة التي لا تزال في reflog). --decorate يُظهر أسماء الفروع والعلامات. استخدمه لفم العلاقات بين الفروع والـ merge points.

git reflog — سجل الحركات

git reflog يُسجّل كل حركة لـ HEAD — حتى commits التي لم يعد لها فرع (dangling commits). منقذ للحياة إن أخطأت:

# عرض سجل كل شيء فعلته git reflog # مثال: لو حذفت commit بالخطأ بـ git reset --hard # 1. شوف reflog لتعرف الـ hash git reflog # انتاج: a1b2c3d HEAD@{2}: commit: Add critical feature # 2. ارجع إليه git reset --hard a1b2c3d
🔍 شرح الكود:

git reflog هو دفتر يوميات Git — يسجل كل حركة: commits, resets, checkouts, merges, rebases. HEAD@{2} يعني "ما كان عليه HEAD قبل حركتين". حتى لو خسرت commit بـ reset --hard أو branch -d، يمكنك استعادته من reflog إذا لم يمضِ وقت طويل (تنظف reflog بعد 90 يوماً).

Detached HEAD — رأس منفصل

حالة Detached HEAD تحدث عندما تنتقل إلى commit معين بدلاً من فرع. تبدو مخيفة لكنها غير خطيرة:

# الدخول في حالة detached HEAD git checkout a1b2c3d # Git يحذّر: You are in 'detached HEAD' state # يمكنك التجربة — إنشاء commits git commit -m "تجربة في detached HEAD" # للخروج — أنشئ فرعاً أو انتقل لفرع موجود git switch -c new-branch-name # أو ببساطة: git switch main
🔍 شرح الكود:

Detached HEAD يعني أن HEAD يشير مباشرة إلى commit وليس إلى فرع. هذا طبيعي — ربما أردت رؤية commit قديم. إذا أنشأت commits في هذه الحالة وخرجت دون ربطها بفرع، Git سيُفقدها (تجميع القمامة). الحل: أنشئ فرعاً بـ git switch -c new-branch قبل الانتقال. reflog يمكنه استعادتها أيضاً.

مشاكل وحلول سريعة

# أردت commit لكن نسيت git add git commit -a -m "Add and commit in one step" # -a = add --all (فقط للملفات المتتبّعة) # أردت تغيير رسالة آخر commit git commit --amend -m "الرسالة الصحيحة" # أضفت ملفاً إلى commit عن طريق الخطأ git reset HEAD^ -- file-to-remove.py git commit --amend # نسيت إضافة ملف إلى commit git add missed-file.py git commit --amend --no-edit # أريد تجاهل ملف تم تتبّعه بالفعل git rm --cached secrets.py # ثم أضف secrets.py إلى .gitignore
🔍 شرح الكود:

git commit -a يضيف ويسجّل في خطوة — لكن فقط للملفات التي Git يتتبّعها أصلاً. --amend يُعدّل آخر commit (لا تفعله إذا كان commit منشوراً على GitHub). git reset HEAD^ -- file يُخرج ملفاً من آخر commit. git rm --cached يمنع Git من تتبع ملف دون حذفه من القرص.

📝 ملخص Git/GitHub

في هذا الدليل تعلمت:

  • مفهوم التحكم بالنسخ (Version Control) وأهمية Git
  • تثبيت Git وتكوين اسم المستخدم والبريد الإلكتروني
  • إنشاء مستودع بـ git init وحالة الملفات (دورة الحياة)
  • إضافة الملفات بـ git add وتسجيلها بـ git commit
  • إنشاء الفروع وإدارتها بـ git branch و git switch
  • دمج الفروع بـ git merge وحل التعارضات
  • العمل مع GitHub: git remote, git push, git pull, git clone
  • سير عمل Pull Requests و Fork
  • إعادة تنظيم التاريخ بـ git rebase و git reset و git revert
  • تجاهل الملفات بـ .gitignore وأفضل الممارسات
  • أدوات استكشاف الأخطاء: git stash, git diff, git reflog
💡 نصيحة ذهبية: لا تخف من Git — كل خطأ تقريباً يمكن إصلاحه. git reflog و git reset و git revert هم أصدقاؤك. commit مبكراً و commit كثيراً. كل commit يجب أن يكون وحدة منطقية واحدة. استخدم فروعاً لكل مهمة. راجع git diff قبل كل commit. و الأهم — مارس Git يومياً حتى يصبح عادة.
⚠️ أخطاء شائعة: 1. استخدام git push --force على فروع مشتركة — هذا يُعيد كتابة التاريخ ويُفسد نسخ زملائك. استخدم --force-with-lease بدلاً منه. 2. commit ملفات كبيرة (datasets, node_modules) — لن يستطيع أحد clone المستودع. أضفها إلى .gitignore و استخدم Git LFS للملفات الكبيرة. 3. تجاهل .gitignore حتى فوات الأوان — من الصعب إزالة ملف بعد تتبّعه. أنشئ .gitignore في أول commit للمشروع. 4. commit على main مباشرة — استخدم الفروع دائماً حتى للتغييرات الصغيرة.
🏋️ تمرين تطبيقي: قم بإنشاء مستودع Git جديد لمشروع بايثون. أنشئ ملف main.py يطبع "Hello, Git!". أنشئ فرعاً باسم feature/calculator وأضف دالة جمع. ادمج الفرع مع main. ادفع المشروع إلى GitHub (أنشئ مستودعاً فارغاً أولاً). ثم افتح Pull Request لتغيير وهمي وادمجه. هذا التمرين يغطي 80% من استخدامات Git اليومية.
