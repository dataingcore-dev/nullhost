Redis — التخزين المؤقت وقوائم الانتظار

قاعدة بيانات في الذاكرة للتخزين المؤقت، الجلسات، والرسائل

← العودة إلى الرئيسية

Redis هو مخزن بيانات في الذاكرة (in-memory) يُستخدم للتخزين المؤقت (caching)، إدارة الجلسات، قوائم الانتظار، والنشر/الاشتراك. سرعته الفائقة تجعله أساسياً في أي تطبيق حديث.

# تثبيت Redis # sudo apt install redis-server (Linux) # brew install redis (macOS) # ثم شغّل: redis-server # مكتبة بايثون pip install redis

بعد التثبيت، تأكد من تشغيل سيرفر Redis على جهازك (redis-server). المكتبة ستتصل به تلقائياً على localhost:6379 وهو المنفذ الافتراضي.

1. العمليات الأساسية

Redis يعمل بنظام key-value — كل قيمة تُخزّن بمفتاح فريد. هذا يشبه القاموس (dict) في بايثون، لكنه مشترك بين كل عمليات التطبيق ويعمل بسرعة فائقة لأنه في الذاكرة. هيا بنا نرى العمليات الأساسية:

import redis r = redis.Redis(host='localhost', port=6379, db=0, decode_responses=True) # String — تخزين key/value r.set('user:1:name', 'Ahmed') print(r.get('user:1:name')) # 'Ahmed' # مع انتهاء صلاحية (TTL) r.setex('session:123', 3600, 'active') # تنتهي بعد ساعة # التحقق من الوجود print(r.exists('user:1:name')) # 1 (True) print(r.ttl('session:123')) # الوقت المتبقي بالثواني
🔍 كيف يعمل؟ redis.Redis(host, port, db, decode_responses) يُنشئ اتصالاً. decode_responses=True يحوّل البايتات إلى نصوص تلقائياً. set يخزّن قيمة، get يسترجعها. setex يخزّن مع وقت انتهاء (TTL) بالثواني — بعد 3600 ثانية (ساعة) تُحذف القيمة تلقائياً. exists يتحقق من وجود المفتاح، ttl يُظهر الوقت المتبقي.

2. List — قوائم الانتظار

القوائم (Lists) في Redis مثالية لبناء قوائم انتظار المهام (Task Queues). يمكنك push عناصر من اليسار أو اليمين، و pop عناصر من أي طرف. مع BLOCKING pop، العامل (worker) ينتظر حتى يأتي عنصر جديد — بدلاً من الاستفسار المتكرر.

# push إلى القائمة r.rpush('queue:tasks', 'task1', 'task2', 'task3') r.lpush('queue:urgent', 'urgent_task') # من البداية # pop من القائمة task = r.lpop('queue:tasks') # من البداية last = r.rpop('queue:tasks') # من النهاية # BLOCKING pop — انتظر حتى يأتي عنصر (مفيد للـ workers) task = r.blpop('queue:tasks', timeout=30) # انتظر 30 ثانية
🔍 كيف يعمل؟ rpush يضيف عناصر إلى نهاية القائمة (يمين)، lpush يضيف إلى البداية (يسار). lpop يسحب من البداية (مثل طابور FIFO)، rpop يسحب من النهاية (مثل Stack LIFO). blpop (Blocking Left Pop) — إذا كانت القائمة فارغة، ينتظر حتى timeout ثانية أو حتى يأتي عنصر جديد. هذا مثالي للـ workers: ينامون بدلاً من إضاعة CPU.

3. Set — مجموعات فريدة

المجموعات (Sets) تخزّن عناصر فريدة بدون ترتيب — تماماً مثل set في بايثون. تدعم عمليات المجموعات الرياضية: التقاطع (intersection)، الاتحاد (union)، والفرق (difference). مثالية للعلامات (tags)، المتابعين، والإحصائيات.

# إضافة عناصر r.sadd('tags:python', 'fastapi', 'pydantic', 'redis') r.sadd('tags:javascript', 'node', 'redis') # عمليات المجموعات print(r.sinter('tags:python', 'tags:javascript')) # تقاطع: {'redis'} print(r.sunion('tags:python', 'tags:javascript')) # اتحاد
🔍 كيف يعمل؟ sadd يضيف عناصر للمجموعة — إذا كان العنصر موجوداً مسبقاً، يتم تجاهله (لا تكرار). sinter يُرجع العناصر المشتركة بين المجموعتين — هنا 'redis' مشترك بين Python و JavaScript. sunion يُرجع كل العناصر من المجموعتين بدون تكرار. هذه العمليات سريعة جداً وتُنفذ على الخادم (server-side).

4. Caching — التخزين المؤقت مع FastAPI

أكثر استخدامات Redis شيوعاً هو التخزين المؤقت (Caching). الفكرة: بدلاً من جلب البيانات من قاعدة البيانات في كل طلب (وهي عملية بطيئة نسبياً)، نخزّن النتيجة في Redis بعد أول طلب. الطلبات التالية تجيب من Redis مباشرة — أسرع بمئات المرات. هذا يُقلل الضغط على قاعدة البيانات ويُسرّع الاستجابة.

from fastapi import FastAPI import redis import json app = FastAPI() cache = redis.Redis(host='localhost', port=6379, db=0) CACHE_TTL = 300 # 5 دقائق @app.get("/products/{product_id}") def get_product(product_id: int): cache_key = f"product:{product_id}" # 1. حاول قراءة من Redis cached = cache.get(cache_key) if cached: return json.loads(cached) # 2. لو مش موجود، جلب من قاعدة البيانات product = get_from_db(product_id) # 3. خزّن في Redis cache.setex(cache_key, CACHE_TTL, json.dumps(product)) return product
🔍 كيف يعمل؟ هذا نمط الـ Cache-Aside: (1) نكوّن مفتاح cache فريد (product:123). (2) نبحث في Redis — إذا وجدنا القيمة، نعيدها فوراً (cache hit). (3) إذا لم نجد (cache miss)، نجلب من قاعدة البيانات. (4) نخزّن النتيجة في Redis مع وقت انتهاء (TTL) حتى تنتهي صلاحيتها تلقائياً بعد 5 دقائق. هذا يضمن أن البيانات لا تصبح قديمة جداً.

5. Session Management — إدارة الجلسات

إدارة جلسات المستخدمين (Sessions) من أشهر استخدامات Redis. بدلاً من تخزين الجلسات في قاعدة البيانات العلائقية (بطيئة) أو في ذاكرة التطبيق (تُفقد عند إعادة التشغيل)، Redis يخزّنها في الذاكرة مع إمكانية انتهاء الصلاحية تلقائياً.

import uuid from datetime import timedelta SESSION_TTL = 86400 # 24 ساعة def create_session(user_id: int) -> str: session_id = str(uuid.uuid4()) session_data = json.dumps({"user_id": user_id, "role": "admin"}) cache.setex(f"session:{session_id}", SESSION_TTL, session_data) return session_id def get_session(session_id: str) -> Optional[dict]: data = cache.get(f"session:{session_id}") return json.loads(data) if data else None
🔍 كيف يعمل؟ create_session: نستخدم uuid لإنشاء معرف فريد للجلسة، نخزّن بيانات الجلسة (user_id, role) كـ JSON مع TTL 24 ساعة — بعدها تنتهي الجلسة تلقائياً. get_session: نبحث عن الجلسة في Redis بـ session_id، إذا وجدناها نُحوّل JSON إلى dict ونعيدها، إذا لا — نعيد None. هذا النمط يُستخدم في معظم تطبيقات الويب الحديثة.

6. Pub/Sub — النشر والاشتراك

Pub/Sub (Publish/Subscribe) هو نظام مراسلة: المُرسِل (publisher) يرسل رسالة إلى قناة (channel) دون معرفة من يستقبلها. المستقبِل (subscriber) يستمع للقناة ويستقبل الرسائل. هذا مثالي للإشعارات الفورية، التحديثات المباشرة، وتوزيع المهام بين الخدمات.

# الناشر (Publisher) r.publish('channel:orders', json.dumps({"order_id": 123})) # المشترك (Subscriber) — يعمل في عملية منفصلة pubsub = r.pubsub() pubsub.subscribe('channel:orders') for message in pubsub.listen(): if message['type'] == 'message': print("New order:", message['data'])
🔍 كيف يعمل؟ publish ترسل رسالة (JSON) إلى قناة 'channel:orders' — أي مشترك يستمع لهذه القناة سيستقبلها. pubsub() يُنشئ كائن اشتراك، subscribe يُضيف القناة، listen() يُعيد رسائل بشكل متتابع — البرنامج ينتظر (blocking) حتى تصل رسالة جديدة. هذا يُستخدم للإشعارات المباشرة: مثلاً عند تقديم طلب جديد، تُرسل إشعاراً لجميع المتصلين.

7. Rate Limiting — تحديد المعدل

منع إساءة الاستخدام (Abuse) ضروري لأي API. Rate Limiting يحدد عدد الطلبات المسموح بها لكل مستخدم في فترة زمنية معينة. Redis مثالي لهذا لأنه سريع جداً ويدعم عمليات INCR الذرية (atomic).

def rate_limit(user_id: int, max_requests: int = 100, window: int = 60) -> bool: key = f"ratelimit:{user_id}" count = r.incr(key) if count == 1: r.expire(key, window) return count <= max_requests # استخدام في FastAPI @app.get("/api/data") def get_data(user_id: int = 1): if not rate_limit(user_id): raise HTTPException(status_code=429, detail="Too many requests") return {"data": "ok"}
🔍 كيف يعمل؟ incr(key) يزيد العداد بمقدار 1 ذرياً (atomic) — حتى لو جاءت طلبات متزامنة، لن يحدث تعارض. إذا كان count == 1 (أول طلب)، نضبط expire لمسح المفتاح بعد window ثانية. return count <= max_requests — إذا تجاوز العدد، نُعيد False ونرفع HTTP 429 (Too Many Requests). هذا النمط يُستخدم في Cloudflare, GitHub, وكل APIs الكبيرة.
