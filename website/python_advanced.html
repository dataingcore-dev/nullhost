Python Advanced — الأنماط، typing، logging، architecture

كتابة كود بايثون احترافي باستخدام type hints، أنماط البرمجة، وتوثيق الأخطاء

← العودة إلى الرئيسية

بعد إتقان أساسيات بايثون، حان الوقت للانتقال إلى المستوى الاحترافي. هذا الفصل يغطي type hints (تحسين قراءة الكود واكتشاف الأخطاء مبكراً)، أنماط العمارة البرمجية (مثل Repository, Service Layer)، و logging (تسجيل الأحداث بدلاً من print).

1. Type Hints — كتابة كود آمن وقابل للصيانة

بايثون لغة ديناميكية، لكن يمكنك إضافة type hints لتحسين القراءة وتمكين أدوات الفحص:

Type Hints تُحدد نوع المتغيرات التي تأخذها الدوال والقيم التي تُرجعها. هذا لا يغيّر سلوك بايثون في التشغيل، لكنه يساعدك (وفريقك) على فهم الكود، ويُمكّن أدوات الفحص مثل mypy من اكتشاف الأخطاء قبل التشغيل. تخيّلها كتوثيق قابل للفحص آلياً.

from typing import List, Dict, Optional, Union, Callable, TypeVar # Type hints أساسية def greet(name: str) -> str: return f"Hello, {name}" # الحاويات def process_ids(ids: List[int]) -> Dict[str, int]: return {"count": len(ids)} # Optional و Union def find_user(uid: int) -> Optional[dict]: return None if uid > 100 else {"id": uid} # Union لأنواع متعددة def parse_value(data: Union[str, bytes]) -> str: return data.decode() if isinstance(data, bytes) else data
🔍 كيف يعمل؟ greet(name: str) -> str: تأخذ الاسم (نص) وتُعيد نصاً. process_ids تأخذ قائمة أرقام (List[int]) وتُعيد قاموساً (Dict[str, int]). Optional[dict] يعني: قد تُعيد dict أو None. Union[str, bytes] يعني: قد تأخذ نصاً أو بايتات — نستخدم isinstance للفحص والتفرع. هذه الأنواع تتيح لأدوات مثل mypy فحص الكود تلقائياً.

TypeVar — أنواع عامة (Generics)

TypeVar يسمح لك بكتابة دوال عامة (Generic) تعمل مع أي نوع، مع الحفاظ على معلومات النوع. بدلاً من فقدان النوع، T يستنتج النوع الفعلي عند الاستخدام.

T = TypeVar('T') def first(items: List[T]) -> T: return items[0] print(first([1, 2, 3])) # int — T يستنتج كـ int print(first(["a", "b"])) # str — T يستنتج كـ str
🔍 كيف يعمل؟ T = TypeVar('T') يُعرِّف متغير نوع عام. الدالة first(items: List[T]) -> T: تأخذ قائمة من أي نوع T وتُعيد عنصراً من نفس النوع T. عندما نمرر [1, 2, 3]، mypy يستنتج أن T = int. عندما نمرر ["a", "b"]، T = str. هذا أفضل من استخدام Any لأننا نحتفظ بمعلومات النوع.
💡 نصيحة: استخدم type hints لكل الدوال. أدوات مثل mypy تفحص الأخطاء قبل التشغيل. شغّل mypy my_file.py لاكتشاف الأخطاء.

2. Logging — تسجيل الأحداث بدلاً من print

استخدام print() لتصحيح الأخطاء مؤقت فقط. في التطبيقات الحقيقية، استخدم logging:

التسجيل (logging) يتيح لك تصنيف الرسائل حسب الأهمية (معلومات، تحذير، خطأ)، تسجيلها في ملفات، وإيقاف/تشغيل مستويات معينة دون حذف الكود. في الإنتاج، logging أفضل من print لأنه يمكنك تتبّع الأخطاء حتى بعد إغلاق الطرفية.

import logging # إعداد logging logging.basicConfig( level=logging.INFO, format='%(asctime)s - %(name)s - %(levelname)s - %(message)s', filename='app.log' # سجل في ملف ) logger = logging.getLogger(__name__) def divide(a: float, b: float) -> float: logger.info(f"Dividing {a} by {b}") if b == 0: logger.error("Division by zero!") raise ValueError("Cannot divide by zero") return a / b logger.debug("Debug message") # التفاصيل الدقيقة logger.info("Info message") # معلومات عامة logger.warning("Warning") "> تحذير logger.error("Error occurred") "> خطأ logger.critical("Critical!") "> خطأ كارثي
🔍 كيف يعمل السطر تلو السطر؟ basicConfig يُهيئ النظام: level=INFO يعني سجّل فقط INFO وما فوق (تجاهل DEBUG). format يُحدد شكل السجل (وقت + اسم + مستوى + رسالة). filename يكتب في ملف بدلاً من الشاشة. getLogger(__name__) يُنشئ مسجّلاً (logger) باسم الملف الحالي — يساعد في تتبّع مصدر الرسالة. كل مستوى له دالة: info, warning, error, critical. في الإنتاج، ارفع المستوى إلى WARNING أو ERROR لتقليل الضوضاء.
💡 مستويات logging: DEBUG (10) < INFO (20) < WARNING (30) < ERROR (40) < CRITICAL (50). ضبط level=logging.INFO يعني تجاهل DEBUG.

Logging مع ملفات دوارة (Rotating)

في التطبيقات الحقيقية، ملفات السجل تكبر بسرعة. RotatingFileHandler يحلّ المشكلة: عندما يصل الملف لحجم معين، يُؤرشفه ويبدأ واحداً جديداً — مع الاحتفاظ بعدد محدد من النسخ الاحتياطية.

from logging.handlers import RotatingFileHandler handler = RotatingFileHandler( 'app.log', maxBytes=1024*1024, backupCount=5 ) logging.basicConfig(level=logging.INFO, handlers=[handler]) # الملف يدور عند 1MB، وآخر 5 نسخ احتياطية
🔍 كيف يعمل؟ RotatingFileHandler يُراقب حجم الملف. عندما يصل لـ maxByte (1MB هنا = 1024*1024)، يُعيد تسمية الملف (app.log → app.log.1) وينشئ app.log جديد. backupCount=5 يحتفظ بآخر 5 نسخ — app.log.1, app.log.2, ... app.log.5. هذا يمنع السجلات من استهلاك كل مساحة القرص.

3. أنماط العمارة البرمجية — Architecture Patterns

مع نمو التطبيق، تحتاج إلى فصل المسؤوليات. إليك أهم الأنماط لتطبيقات بايثون:

نمط Repository — فصل منطق البيانات

Repository Pattern يفصل منطق الوصول إلى البيانات (SQL, API, ملفات) عن باقي التطبيق. بدلاً من كتابة استعلامات SQL في كل مكان، نُعرِّف واجهة (Interface) ونتنفيذها في كلاسات منفصلة. هذا يسهّل تغيير قاعدة البيانات لاحقاً واختبار الكود.

from abc import ABC, abstractmethod from typing import List, Optional # واجهة抽象 للمستودع class UserRepository(ABC): def get_by_id(self, uid: int) -> Optional[dict]: ... def get_all(self) -> List[dict]: ... def save(self, user: dict) -> int: ... # تنفيذ مع PostgreSQL class PostgresUserRepository(UserRepository): def __init__(self, connection): self.conn = connection def get_by_id(self, uid: int) -> Optional[dict]: cursor = self.conn.cursor() cursor.execute("SELECT * FROM users WHERE id = %s", (uid,)) row = cursor.fetchone() return dict(row) if row else None def get_all(self) -> List[dict]: cursor = self.conn.cursor() cursor.execute("SELECT * FROM users") return [dict(row) for row in cursor.fetchall()] def save(self, user: dict) -> int: cursor = self.conn.cursor() cursor.execute( "INSERT INTO users (name, email) VALUES (%s, %s) RETURNING id", (user['name'], user['email']) ) self.conn.commit() return cursor.fetchone()[0]
🔍 كيف يعمل؟ UserRepository هو كلاس مجرّد (ABC) يُعرِّف الواجهة: get_by_id, get_all, save — دون تنفيذ. PostgresUserRepository يُنفّذ هذه الدوال باستخدام psycopg2. الميزة: يمكنك إنشاء InMemoryUserRepository للاختبار، أو MongoUserRepository لتغيير قاعدة البيانات — كلها تتبع نفس الواجهة. التطبيق لا يعرف أي نوع مستودع يستخدم (Dependency Injection).

نمط Service Layer — منطق الأعمال

Service Layer يحتوي على منطق الأعمال (Business Logic) — القواعد التي تميّز تطبيقك. هو الطبقة الوسطى بين واجهة المستخدم (API/UI) وطبقة البيانات (Repository). هنا نتحقق من صحة البيانات، نسجّل الأحداث، ونطبق قواعد العمل.

class UserService: def __init__(self, repo: UserRepository, logger: logging.Logger): self.repo = repo self.logger = logger def register_user(self, name: str, email: str) -> int: if not email or '@' not in email: raise ValueError("Invalid email") user_id = self.repo.save({"name": name, "email": email}) self.logger.info(f"User registered: {user_id}") return user_id def get_user(self, uid: int) -> dict: user = self.repo.get_by_id(uid) if not user: raise ValueError(f"User {uid} not found") return user
🔍 كيف يعمل؟ UserService يستقبل repository و logger عن طريق Dependency Injection — لا ينشئهما بنفسه (مرونة أعلى). register_user: يتحقق من صحة الإيميل (تحتوي @)، ثم يحفظ عبر repo، ويسجّل العملية. get_user: يبحث عن المستخدم، وإذا لم يُوجد يرفع ValueError. Service Layer لا يعرف تفاصيل قاعدة البيانات — هذا مسؤولية Repository.

نمط Builder — بناء كائنات معقدة

Builder Pattern يسمح ببناء كائن معقّد خطوة بخطوة بطريقة مقروءة. كل دالة تُعدّل الكائن وتُعيد self (نفس الكائن) — هذا يُسمى Method Chaining. النتيجة: كود يشبه الجمل الإنجليزية.

class QueryBuilder: def __init__(self): self._select = "*" self._from = "" self._where = [] self._order_by = "" def select(self, fields: str): self._select = fields; return self def from_table(self, table: str): self._from = table; return self def where(self, condition: str): self._where.append(condition); return self def build(self) -> str: query = f"SELECT {self._select} FROM {self._from}" if self._where: query += " WHERE " + " AND ".join(self._where) return query # استخدام query = QueryBuilder().select("id, name").from_table("users").where("age > 18").build() print(query) # SELECT id, name FROM users WHERE age > 18
🔍 كيف يعمل؟ كل دالة (select, from_table, where) تُحدّث حالة الكائن وتُعيد self — مما يسمح بالتسلسل (chaining). build() تجمّع الاستعلام النهائي. هذا أفضل من تمرير 10 وسائط لدالة constructor. النمط مستخدم في مكتبات معروفة: SQLAlchemy, Django ORM, و FastAPI (في إعداد المسارات).

4. Dataclasses — بيانات نظيفة بدون boilerplate

كتابة كلاس عادي لتخزين بيانات يتطلب كتابة __init__ و __repr__ و __eq__ يدوياً — الكثير من الكود المتكرر (boilerplate). @dataclass يُولّد كل هذا تلقائياً. فقط عرّف الحقول (fields) والنوع، واترك الباقي لبايثون.

from dataclasses import dataclass, field @dataclass class User: id: int name: str email: str age: int = 0 tags: List[str] = field(default_factory=list) def is_adult(self) -> bool: return self.age >= 18 user = User(id=1, name="Ahmed", email="a@b.com", age=25) print(user.is_adult()) # True
🔍 كيف يعمل؟ @dataclass يُنشئ تلقائياً: __init__ (id, name, email, age, tags)، __repr__ (تمثيل readable)، __eq__ (مقارنة). age: int = 0 قيمة افتراضية. tags: القائمة تحتاج default_factory=list (وليس =[] مباشرة) لأن القيم الافتراضية القابلة للتغيير (mutable) تسبب مشاكل في بايثون. يمكنك إضافة دوال عادية (is_adult) بجانب الحقول.

5. Context Managers — إدارة الموارد

Context Managers (مديري السياق) يضمنون فتح وإغلاق الموارد (ملفات، اتصالات قاعدة بيانات، locks) بشكل آمن — حتى لو حدث خطأ. تستخدمهم مع جملة with. يمكنك تنفيذهم بطريقتين: كلاس مع __enter__ و __exit__، أو decorator @contextmanager الأسهل.

from contextlib import contextmanager # باستخدام الكلاس class DatabaseConnection: def __enter__(self): self.conn = "open_connection" return self.conn def __exit__(self, exc_type, exc_val, exc_tb): print("Closing connection") with DatabaseConnection() as conn: print("Working with", conn) # باستخدام contextmanager (أسهل) @contextmanager def open_file(path: str, mode: str = 'r'): file = open(path, mode) try: yield file finally: file.close() with open_file("test.txt", "w") as f: f.write("Hello")
🔍 كيف يعمل؟ الطريقة الأولى (كلاس): __enter__ يُنفّذ عند الدخول إلى with ويُعيد المورد. __exit__ يُنفّذ عند الخروج (حتى مع الأخطاء) ويُغلق المورد. الطريقة الثانية (@contextmanager): الدالة تفتح المورد، yield يُسلّمه لكتلة with، finally يُغلقه — أسهل وأقل كود. كلتاهما تضمن أن المورد سيُغلق حتى لو حدث خطأ داخل with.
