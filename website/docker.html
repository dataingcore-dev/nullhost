Docker — الحاويات والنشر

بناء، تشغيل، ونشر التطبيقات باستخدام الحاويات

← العودة إلى الرئيسية

Docker يسمح لك بتغليف التطبيق مع جميع اعتمادياته في حاوية خفيفة وقابلة للنقل. الحاويات تعمل بنفس الطريقة في أي بيئة — من لابتوبك إلى السيرفر.

تثبيت Docker:
sudo apt install docker.io (Linux) | brew install --cask docker (macOS) | تحميل من docker.com (Windows)

1. Dockerfile — وصف الحاوية

Dockerfile هو ملف نصي يصف كيفية بناء صورة الحاوية (image). كل سطر فيه هو تعليمة (instruction) تُنفّذ بالترتيب. الصورة الناتجة تحتوي على كل ما تحتاجه لتشغيل التطبيق: نظام تشغيل، مكتبات، كود، وإعدادات.

# Dockerfile — لتطبيق FastAPI FROM python:3.11-slim WORKDIR /app COPY requirements.txt . RUN pip install --no-cache-dir -r requirements.txt COPY . . EXPOSE 8000 CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
🔍 كيف يعمل؟ FROM: نبدأ من صورة أساسية (python:3.11-slim — نسخة مصغرة من بايثون). WORKDIR: نحدد مجلد العمل داخل الحاوية. COPY: ننسخ الملفات من جهازك للحاوية (ننسخ requirements أولاً ثم نشغل pip للاستفادة من الـ caching). RUN: ننفذ أوامر (تثبيت المكتبات). EXPOSE: نُعلن أن التطبيق يستخدم المنفذ 8000. CMD: الأمر الذي يُشغّل عند بدء الحاوية.
# requirements.txt fastapi uvicorn redis psycopg2-binary

2. بناء وتشغيل الحاوية

بعد كتابة Dockerfile، نستخدم أوامر Docker لبناء الصورة وتشغيل الحاوية. عملية البناء تُنشئ صورة (image) — وهي قالب ثابت. تشغيل الحاوية (container) يُنشئ نسخة قابلة للتشغيل من تلك الصورة.

# بناء الصورة docker build -t my-fastapi-app . # تشغيل الحاوية docker run -d --name my-app -p 8000:8000 my-fastapi-app # أوامر مفيدة docker ps # الحاويات العاملة docker ps -a # جميع الحاويات docker logs my-app # السجلات docker exec -it my-app /bin/bash # دخول الحاوية docker stop my-app # إيقاف docker rm my-app # حذف
🔍 كيف يعمل؟ docker build -t my-fastapi-app . يبني الصورة من الـ Dockerfile في المجلد الحالي (-t للاسم). docker run -d يشتغل الحاوية في الخلفية (detached) مع تعيين اسم (--name) وربط المنفذ 8000 المحلي بـ 8000 في الحاوية. docker exec -it يدخل الحاوية بتفاعل (bash). docker logs يعرض السجلات — مفيد جداً لتصحيح الأخطاء.

3. Docker Compose — تطبيقات متعددة الخدمات

عندما يحتاج تطبيقك إلى أكثر من خدمة (مثل FastAPI + PostgreSQL + Redis)، إدارة كل حاوية على حدة تصبح معقدة. Docker Compose يسمح لك بتعريف كل الخدمات في ملف YAML واحد، وتشغيلها بأمر واحد.

# docker-compose.yml version: '3.8' services: app: build: . ports: - "8000:8000" depends_on: - db - redis environment: - DATABASE_URL=postgresql://user:pass@db:5432/mydb - REDIS_URL=redis://redis:6379/0 db: image: postgres:15 environment: POSTGRES_USER: user POSTGRES_PASSWORD: pass POSTGRES_DB: mydb volumes: - postgres_data:/var/lib/postgresql/data redis: image: redis:7-alpine volumes: postgres_data:
🔍 كيف يعمل؟ نعرّف 3 خدمات: app (تطبيقنا)، db (PostgreSQL)، redis (Redis). depends_on يضمن أن db و redis يبدآن قبل app. environment يُمرّر متغيرات البيئة — لاحظ أن اسم المضيف (host) هو اسم الخدمة (db, redis). volumes تحفظ بيانات PostgreSQL خارج الحاوية حتى لا تُفقد عند إعادة التشغيل.
# تشغيل كل الخدمات docker-compose up -d # إيقاف docker-compose down # إعادة بناء docker-compose up -d --build

docker-compose up -d يبني ويشغّل كل الخدمات في الخلفية. docker-compose down يوقف ويحذف كل الحاويات (لكن يحتفظ بالبيانات لأنها في volume). --build يُعيد بناء الصور قبل التشغيل — استخدمه عندما تغيّر الكود.

4. أفضل الممارسات

  • Multi-stage builds: استخدم مراحل متعددة لتقليل حجم الصورة
  • .dockerignore: استبعد الملفات غير الضرورية (مثل __pycache__, .git)
  • استخدم images رسمية: مثل python:3.11-slim بدلاً من python:3.11 (أصغر حجماً)
  • لا تشغل كـ root: أضف مستخدم غير privileged
# Multi-stage build مثال # المرحلة الأولى: بناء FROM python:3.11-slim AS builder COPY requirements.txt . RUN pip install --user -r requirements.txt # المرحلة الثانية: تشغيل FROM python:3.11-slim COPY --from=builder /root/.local /root/.local COPY . . ENV PATH=/root/.local/bin:$PATH USER 1000 CMD ["uvicorn", "main:app", "--host", "0.0.0.0"]
🔍 كيف يعمل؟ Multi-stage build: المرحلة الأولى (builder) تُثبّت المكتبات، والمرحلة الثانية تنسخ منها فقط الملفات الناتجة (ليس الـ layer cache كاملاً). هذا يُقلص حجم الصورة النهائية بشكل كبير — لأن الأدوات الوسيطة (pip cache, compiler) لا تنتقل للمرحلة الثانية. USER 1000 يُشغّل التطبيق بمستخدم غير root للأمان.

5. Docker Networking

الشبكات في Docker تسمح للحاويات بالتواصل مع بعضها البعض. كل حاوية على نفس الشبكة يمكنها الوصول للأخرى باستخدام اسم الحاوية كـ hostname — هذا أفضل من استخدام عناوين IP التي قد تتغير.

# إنشاء شبكة docker network create my-network # تشغيل حاويات على نفس الشبكة docker run -d --name redis --network my-network redis:7-alpine docker run -d --name app --network my-network -p 8000:8000 my-fastapi-app # في الكود، استخدم اسم الحاوية كـ hostname # redis_client = redis.Redis(host='redis', port=6379)

باستخدام --network، كل الحاويات على نفس الشبكة يمكنها رؤية بعضها بأسمائها. في كود بايثون، استخدم host='redis' (اسم الحاوية) بدلاً من localhost. هذا النمط يُستخدم تلقائياً في Docker Compose.

6. Docker Volumes — البيانات الدائمة

الحاويات ephemeral — أي تغيير داخل الحاوية يُفقد عند حذفها. Volumes تحل هذه المشكلة بتخزين البيانات خارج الحاوية على جهاز المضيف. هذا ضروري لقواعد البيانات، الملفات، والسجلات.

# Named volume (يديره Docker) docker run -v postgres_data:/var/lib/postgresql/data postgres:15 # Bind mount (مجلد محلي) docker run -v $(pwd)/data:/app/data my-app # مشاركة الملفات بين الحاوية والمضيف docker run -v $(pwd)/logs:/app/logs my-app
🔍 كيف يعمل؟ Named volume (postgres_data): Docker يُدير المجلد — أين يقع على المضيف لا يهمك، فقط استخدم الاسم. Bind mount ($(pwd)/data): تربط مجلداً محدداً على جهازك بمجلد في الحاوية — مفيد للتطوير (تعدّل الكود على جهازك والحاوية تراها فوراً). الفرق: named volumes أسهل للإدارة، bind mounts أسرع للتطوير.

7. التمارين

تمرين 1: حاوية FastAPI
أنشئ Dockerfile لتطبيق FastAPI بسيط، ابنِ الصورة، وشغّلها على منفذ 8000.
تمرين 2: Docker Compose
أضف PostgreSQL و Redis إلى تطبيقك باستخدام docker-compose.yml.
تمرين 3: Multi-stage
حول Dockerfile الخاص بك إلى multi-stage build وقلل حجم الصورة إلى أقل من 200MB.

📝 ملخص Docker

  • Dockerfile: يصف كيفية بناء الصورة
  • docker build / run: بناء وتشغيل الحاويات
  • Docker Compose: تشغيل عدة خدمات معاً
  • Volumes: حفظ البيانات خارج الحاوية
  • Networks: ربط الحاويات ببعضها
